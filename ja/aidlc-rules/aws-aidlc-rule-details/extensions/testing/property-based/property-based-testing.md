# プロパティベーステストルール

## 概要

これらのプロパティベーステスト（property-based testing、PBT）ルールは、AI-DLC(AI駆動開発ライフサイクル)の該当フェーズ全体に横断的に適用される制約です。これらのルールは、識別可能なプロパティを持つコードがプロパティベースの手法でテストされることを保証し、従来の具体例ベースのテストを置き換えるのではなく補完します。

プロパティベーステストは、すべての有効な入力に対して成立しなければならない不変条件を定義し、フレームワークを使用してランダムな入力を生成し、反例を探索します。失敗が見つかった場合、フレームワークは入力を最小の再現ケースに縮小（シュリンク）します。このアプローチは、具体例ベースのテストが見落としがちなエッジケースや微妙なバグを発見します。

**適用方法**: 各適用対象ステージにおいて、モデルはステージ完了メッセージをユーザーに提示する前に、これらのルールへの準拠を確認しなければなりません。

### ブロッキングPBT所見の動作

**ブロッキングPBT所見**とは、以下を意味します：
1. 該当の所見は、PBTルールIDおよび説明を含む「PBT Findings」セクションの下で、ステージ完了メッセージに必ず列挙されなければなりません。
2. すべてのブロッキング所見が解決されるまで、ステージは「次のステージへ進む」オプションを提示してはなりません。
3. モデルは「変更を要求する」オプションのみを提示し、変更すべき点を明確に説明しなければなりません。
4. 該当の所見は、PBTルールID、説明、およびステージコンテキストを含めて `aidlc-docs/audit.md` に記録されなければなりません。

PBTルールが現在のプロジェクトまたは単位に適用されない場合（例：ステートフルなコンポーネントが存在しない場合のPBT-06）、コンプライアンス要約において **N/A** とマークしてください。これはブロッキング所見ではありません。

### デフォルトの適用

本ドキュメントのすべてのルールは、デフォルトで**ブロッキング**です。任意のルールの検証基準が満たされない場合、それはブロッキングPBT所見となります — 上記のブロッキング所見の動作に従ってください。

### 部分適用モード

ユーザーがオプトイン時に**部分適用**を選択した場合、PBT-02、PBT-03、PBT-07、PBT-08、およびPBT-09のみが適用されます。その他のすべてのルールは推奨事項（非ブロッキング）として扱われます。`aidlc-docs/aidlc-state.md` の `## Extension Configuration` の下に適用モードを記録してください。

### 検証基準のフォーマット

本ドキュメントの検証項目は、コンプライアンスチェックを記述したプレーンな箇条書きです。各項目は、レビュー時に準拠・非準拠として評価されます。

## ルール PBT-01: 設計時のプロパティの特定

**ルール**: ビジネスロジック、データ変換、またはアルゴリズム操作を含むすべての単位(unit)は、機能設計（Functional Design）ステージ中にテスト可能なプロパティについて分析されなければなりません。分析は以下のプロパティカテゴリのどれが該当するかを特定しなければなりません：

| カテゴリ | 説明 | 例 |
|---|---|---|
| Round-trip | 逆操作を組み合わせると元の値が得られる | serialize → deserialize = identity |
| Invariant | 変換がある測定可能な特性を保持する | sort は コレクションのサイズと要素を保持する |
| Idempotence | 操作を2回適用すると1回と同じ結果になる | dedup(dedup(list)) = dedup(list) |
| Commutativity | 操作の順序が異なっても同じ結果が得られる | add(a, b) = add(b, a) |
| Oracle | リファレンス実装または簡略化されたモデルで結果を検証できる | 最適化アルゴリズム vs 総当たり法 |
| Induction | 小さい入力で証明されたプロパティが大きい入力にも拡張される | 再帰構造、分割統治 |
| Easy verification | 結果の計算は難しいが検証は容易 | 迷路ソルバーの出力を辿って検証できる |

特定されたプロパティは、その単位の機能設計成果物に文書化されなければならず、PBTテスト要件としてコード生成に引き継がれなければなりません。

**検証**:
- 機能設計成果物に、コンポーネントごとに特定されたプロパティを列挙した「Testable Properties」セクションが含まれていること
- 特定された各プロパティが上記カテゴリのいずれかを参照していること
- テスト可能なプロパティが特定されなかったコンポーネントに、「No PBT properties identified」と簡単な理由が明示されていること
- プロパティリストがコード生成計画時に参照されること

## ルール PBT-02: ラウンドトリッププロパティ

**ルール**: 論理的な逆操作を持つすべての操作は、ラウンドトリップを検証するプロパティベーステストを持たなければなりません。対象は以下を含みますが、これに限定されません：
- シリアライゼーション / デシリアライゼーション（JSON、XML、Protobuf、バイナリフォーマット）
- エンコード / デコード（Base64、URLエンコード、圧縮）
- パース / フォーマット（日付パース、数値フォーマット、構造化入力でのテンプレートレンダリング）
- 暗号化 / 復号（キーが利用可能な場合）
- データベースへの書き込み / 読み取り（I/O自体ではなくデータ変換レイヤーに対して）
- すべての有効な `x` に対して `f_inverse(f(x)) = x` となる関数のペア

プロパティベーステストは、ドメインに適切なジェネレーター（PBT-07参照）を使用してランダムな有効入力を生成し、ラウンドトリップが元の入力と等しい値を生成することを検証しなければなりません。

**検証**:
- すべてのシリアライゼーション/デシリアライゼーションのペアにラウンドトリッププロパティテストがあること
- すべてのエンコード/デコードのペアにラウンドトリッププロパティテストがあること
- すべてのパース/フォーマットのペアにラウンドトリッププロパティテストがあること（または変換が非可逆である理由を文書化していること）
- ラウンドトリップテストがハードコードされた例ではなく生成された入力を使用すること
- 非可逆変換（例：精度損失を伴う浮動小数点フォーマット）が許容可能な偏差を文書化し、許容範囲内でテストすること

## ルール PBT-03: 不変条件プロパティ

**ルール**: 文書化された不変条件を持つ関数は、生成された入力に対してそれらの不変条件が成立することを検証するプロパティベーステストを持たなければなりません。一般的な不変条件には以下が含まれます：
- **サイズの保持**: 出力コレクションが入力と同じサイズを持つ（例：map、sort）
- **要素の保持**: 出力が入力と完全に同じ要素を含む（順序は変わる場合がある）（例：sort、shuffle）
- **順序の保証**: 出力が順序制約を満たす（例：sortが非減少順を生成する）
- **範囲制約**: 出力値が定義された範囲内に収まる（例：normalizeが[0, 1]の値を生成する）
- **型の保持**: すべての有効な入力に対して出力の型が期待される型と一致する
- **ビジネスルールの不変条件**: 常に成立しなければならないドメイン固有のルール（例：「有効なトランザクション後に口座残高がマイナスにならない」、「割引が商品価格を超えない」）

**検証**:
- 文書化された各不変条件に対応するプロパティベーステストがあること
- 不変条件テストが境界値を含む幅広い入力範囲を生成すること
- 機能設計で特定されたビジネスルールの不変条件がPBTでカバーされていること
- 不変条件テストが具体例ベースのテストからの正確な検証を複製しないこと — 特定のケースではなく一般的なルールをテストすること

---

## ルール PBT-04: 冪等性プロパティ

**ルール**: 冪等性を主張または要求するすべての操作は、それを証明するプロパティベーステストを持たなければなりません。テストは、すべての有効な生成入力に対して `f(f(x)) = f(x)` を検証しなければなりません。対象は以下に適用されます：
- 冪等として文書化されたAPIエンドポイント（PUT、DELETE）
- データの正規化またはサニタイズ関数
- キャッシュの投入操作
- 重複排除ロジック
- 設定の適用（設定を2回適用してもステートが変わらないこと）
- 少なくとも1回の配信システムにおけるメッセージ処理

**検証**:
- 冪等として文書化されたすべての操作に `f(f(x)) = f(x)` を検証するPBTがあること
- 冪等性テストがドメインに適切なジェネレーターを使用すること（プリミティブ型のみではないこと）
- ステートフルな操作では、テストが単一適用と繰り返し適用後の観察可能なステートの等価性を検証すること

---

## ルール PBT-05: オラクルとモデルベーステスト

**ルール**: リファレンス実装、簡略化されたモデル、または既知の正しいアルゴリズムが存在する場合、プロパティベーステストはテスト対象のシステムをオラクルと比較しなければなりません。対象は以下に適用されます：
- 既知の総当たり法を置き換える最適化アルゴリズム
- レガシー実装を置き換えるリファクタリングされたコード
- 逐次バージョンと比較される並列/並行実装
- 既知のアルゴリズム（ソート、検索、グラフ探索）のカスタム実装
- リファレンスデータベースと比較される新しいクエリエンジン

プロパティベーステストはランダムな有効入力を生成し、すべての生成された入力に対してテスト対象のシステムがオラクルと等価な結果を生成することを検証しなければなりません。

**検証**:
- リファレンス実装が存在する（または簡単に書ける）場合にオラクルPBTが存在すること
- オラクルテストが通常、境界、および敵対的なケースをカバーする多様な入力を生成すること
- 等価性が正確に定義されていること（完全一致、構造的等価、または文書化された許容範囲）
- オラクルが存在しない場合、このルールは理由を添えてN/Aとしてマークされること

---

## ルール PBT-06: ステートフルプロパティテスト

**ルール**: 可変ステートを管理するコンポーネントは、ステートフルプロパティテストについて評価されなければなりません。ステートフルPBTはシステムに対してランダムなコマンドシーケンス（操作）を生成し、各ステップ後に不変条件が成立することを検証します。対象は以下に適用されます：
- インメモリキャッシュとデータストア
- ステートマシンとワークフローエンジン
- キューとバッファの実装
- セッション管理システム
- ショッピングカート、注文パイプライン、および類似のステートフルなビジネスオブジェクト
- 操作の結果が以前の操作に依存するすべてのコンポーネント

ステートフルPBTは以下を行わなければなりません：
- テスト対象のシステムを反映する簡略化されたモデル（リファレンスステート）を定義すること
- 有効なコマンドのランダムシーケンスを生成すること（追加、削除、更新、クエリなど）
- 各コマンドを実際のシステムとモデルの両方に対して実行すること
- 各コマンド後に、システムとモデル間の観察可能なステートまたはクエリ結果が一致することを検証すること
- 空のシーケンスを含む様々な長さのシーケンスをテストすること

**検証**:
- 機能設計で特定されたステートフルなコンポーネントにステートフルPBTがあるか、適用できない理由が文書化されていること
- 比較のための簡略化されたモデルが定義されていること
- コマンドジェネレーターが現実的なパラメーター分布を持つ有効な操作シーケンスを生成すること
- シーケンスの最後だけでなく、各コマンド後に不変条件がチェックされること
- ステートフルなコンポーネントが存在しない場合、このルールはN/Aとしてマークされること

---

## ルール PBT-07: ジェネレーターの品質

**ルール**: プロパティベーステストは、単なるプリミティブ型ではなく、現実的で構造化された入力を生成するドメイン固有のジェネレーターを使用しなければなりません。品質の低いジェネレーター（例：メールフィールドに対するランダム文字列、年齢フィールドに対する制限のない整数）は無意味なテストケースを生成し、実際のバグを見逃します。

ジェネレーターの要件：
- **ドメイン型**: ビジネス制約を尊重するドメインオブジェクト（例：User、Order、Transaction）のカスタムジェネレーターを作成しなければなりません（有効なメールフォーマット、正の金額、有効な日付範囲）
- **制約付きプリミティブ**: 数値ジェネレーターはドメインが要求する場合に現実的な範囲に制約されなければなりません
- **構造化データ**: 複雑な入力（ネストされたオブジェクト、ドメインオブジェクトのリスト）のジェネレーターは構造的に有効なデータを生成しなければなりません
- **エッジケースの包含**: ジェネレーターは通常の値とともに境界値（空のコレクション、ゼロ、最大値、Unicode文字列）を含むように設定すべきです
- **再利用可能性**: ドメインジェネレーターはテストファイル全体で複製するのではなく、再利用可能なテストユーティリティとして定義すべきです

**検証**:
- ドメイン型のパラメーターに対して生のプリミティブジェネレーターのみ（例：`st.integers()` のみ）を使用するPBTが存在しないこと
- PBTで使用されるドメインオブジェクトのカスタムジェネレーターが存在すること
- ジェネレーターが文書化されたビジネス制約を尊重すること（例：正の金額、有効なフォーマット）
- 同じドメイン型を複数のテストが共有する場合、ジェネレーターの定義が集中管理され再利用可能であること

---

## ルール PBT-08: シュリンキングと再現性

**ルール**: すべてのプロパティベーステストはシュリンキングと決定論的な再現性をサポートしなければなりません。

- **シュリンキング**: プロパティが失敗した場合、PBTフレームワークは失敗した入力を最小の再現ケースに自動的に縮小しなければなりません。技術的な理由が文書化されない限り（例：外部サービス呼び出しを伴う統合テストでシュリンキングが非互換）、テストはフレームワークのシュリンキングメカニズムを無効化またはバイパスしてはなりません。
- **再現性**: すべてのPBT実行はシード値によって再現可能でなければなりません。失敗時には、正確な失敗シナリオを再現できるよう、シードをログに記録しなければなりません。CI設定は、決定論的な実行のために固定シードを使用するか、失敗後の再現のためにすべての実行でランダムシードをログに記録しなければなりません。
- **CI統合**: PBTはプロジェクトのCIパイプラインに含めなければなりません。不安定なPBT失敗（コード変更なしに再試行で合格するテスト）は、抑制するのではなく調査しなければなりません。

**検証**:
- PBTフレームワークのシュリンキングが有効であること（オーバーライドまたは無効化されていないこと）
- 失敗時のテスト出力にシード値と縮小された最小失敗入力が含まれること
- CI設定がすべてのPBT実行でシードをログに記録するか、固定シードを使用すること
- 文書化された正当な理由なしにCIからPBTが除外されていないこと
- 不安定なPBT失敗が黙って再試行されるのではなく追跡・調査されること

---

## ルール PBT-09: フレームワークの選択

**ルール**: プロジェクトは、主要言語に対して適切なプロパティベーステストフレームワークを選択・設定しなければなりません。フレームワークは以下をサポートしなければなりません：
- ドメイン型に対するカスタムジェネレーター/ストラテジー
- 失敗ケースの自動シュリンキング
- シードベースの再現性
- プロジェクトの既存テストランナーとの統合

言語別の推奨フレームワーク（非網羅的）：

| 言語 | フレームワーク | 備考 |
|---|---|---|
| Python | Hypothesis | 成熟しており、優れたシュリンキング、Django統合 |
| JavaScript / TypeScript | fast-check | Jest、Vitest、Mochaと統合 |
| Java | jqwik | JUnit 5統合、ステートフルテストサポート |
| Kotlin | Kotest Property Testing | Kotestフレームワーク統合 |
| Scala | ScalaCheck | SBT統合、広く採用されている |
| Rust | proptest | マクロベース、優れたシュリンキング |
| Go | rapid | 軽量、イディオマティックなGo |
| Haskell | QuickCheck | オリジナルのPBTフレームワーク |
| C# / .NET | FsCheck | xUnit、NUnitと動作 |
| Erlang / Elixir | PropEr / StreamData | OTP対応、ステートフルテスト |

選択されたフレームワークはテックスタックの決定事項に文書化され、プロジェクトの依存関係として含まれなければなりません。

**検証**:
- PBTフレームワークが選択され、テックスタックの決定事項に文書化されていること
- フレームワークがプロジェクトの依存関係に含まれていること（package.json、pom.xml、requirements.txtなど）
- フレームワークがカスタムジェネレーター、シュリンキング、およびシードベースの再現性をサポートすること
- プロジェクトが複数の言語を使用する場合、PBT適用可能なコードを持つ各言語にフレームワークが選択されていること

---

## ルール PBT-10: 補完的なテスト戦略

**ルール**: プロパティベーステストは具体例ベースのテストを補完しなければならず、置き換えてはなりません。2つのアプローチは異なる目的を果たします：

- **具体例ベースのテスト**: 明示的な期待値を持つ特定の既知シナリオ、回帰ケース、およびビジネスクリティカルなエッジケースを文書化します。具体的な動作の実行可能なドキュメントとして機能します。
- **プロパティベーステスト**: 広い入力空間にわたる一般的な不変条件を検証します。未知のエッジケースを発見し、プロパティが普遍的に成立することを検証します。

要件：
- ユーザーストーリーまたは要件で特定されたビジネスクリティカルなシナリオは、PBTが同じプロパティをカバーしていても、明示的な具体例ベースのテストを持たなければなりません
- PBTはビジネスクリティカルなパスの唯一のテストであってはなりません — 主要シナリオの期待される動作を固定するために少なくとも1つの具体例ベースのテストが必要です
- PBTが失敗ケースを発見した場合、縮小された最小例を永続的な具体例ベースの回帰テストとして追加すべきです
- テスト文書は具体例ベースのテストとプロパティベーステストを明確に区別しなければなりません（別々のテストファイル、テストクラス、または明確に命名されたテスト関数）

**検証**:
- ビジネスクリティカルなパスに具体例ベースとプロパティベースの両方のテストがあること
- PBTが重要な機能の唯一のテストカバレッジとして使用されていないこと
- テストファイルまたはテストクラスがPBTと具体例ベースのテストを明確に分離またはラベル付けしていること
- PBTで発見された失敗からの回帰テストが永続的な具体例ベースのテストとして取り込まれていること

---

## 適用の統合

これらのルールは、以下のAI-DLCステージに適用される横断的な制約です：

| ステージ | 適用ルール | 適用方法 |
|---|---|---|
| Functional Design | PBT-01 | プロパティの特定が設計成果物に含まれなければならない |
| NFR Requirements | PBT-09 | フレームワークの選択がテックスタックの決定事項に含まれなければならない |
| Code Generation (Planning) | PBT-01 〜 PBT-10 | コード生成計画に特定されたプロパティのPBTテストステップを含めなければならない |
| Code Generation (Generation) | PBT-02 〜 PBT-08、PBT-10 | 生成されたテストに具体例ベースのテストとともにPBTを含めなければならない |
| Build and Test | PBT-08 | テスト実行手順にシードロギングとCI統合を含むPBTを含めなければならない |

各適用対象ステージにおいて：
- 生成された成果物に対してすべてのPBTルールの検証基準を評価すること
- ステージ完了サマリーの「PBT Compliance」セクションに、各ルールを準拠・非準拠・N/Aとして列挙すること
- いずれかのルールが非準拠の場合、それはブロッキングPBT所見です — 概要で定義されたブロッキング所見の動作に従ってください
- 設計文書およびテスト手順にPBTルールの参照を含めること

## 付録：プロパティカテゴリクイックリファレンス

Functional Design時にプロパティを特定する開発者とAIモデルのために（PBT-01）：

| パターン名 | 正式な用語 | テストの形式 | 使用するタイミング |
|---|---|---|---|
| There and back again | 可逆関数 | `f_inv(f(x)) == x` | シリアライゼーション、エンコード、パース |
| Some things never change | 不変条件 | `measure(f(x)) == measure(x)` | ソート、マップ、フィルター、変換 |
| The more things change, the more they stay the same | 冪等性 | `f(f(x)) == f(x)` | 正規化、重複排除、キャッシュ書き込み |
| Different paths, same destination | 可換性 | `f(g(x)) == g(f(x))` | 算術、集合操作、独立した変換 |
| Solve a smaller problem first | 構造的帰納法 | `x` に対するプロパティが `x + element` にも成立 | 再帰構造、リスト、ツリー |
| Hard to prove, easy to verify | 検証 | `verify(solve(x)) == true` | ソルバー、オプティマイザー、検索アルゴリズム |
| The test oracle | リファレンス比較 | `f(x) == oracle(x)` | 最適化 vs 総当たり法、リファクタリング vs レガシー |

出典：プロパティカテゴリの分類体系はScott Wlaschinの「Choosing properties for property-based testing」（[fsharpforfunandprofit.com](https://fsharpforfunandprofit.com/posts/property-based-testing-2/)）から適用。
