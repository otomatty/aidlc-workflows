# コード生成 - 詳細手順

## 概要
このステージでは、各作業単位(unit of work)に対してコードを生成する。2つの統合されたパートで構成される:
- **パート 1 - 計画**: 明示的な手順を含む詳細なコード生成計画の作成
- **パート 2 - 生成**: 承認された計画を実行し、コード・テスト・成果物を生成する

**注意**: ブラウンフィールドプロジェクトの場合、「生成」とは既存ファイルを複製するのではなく、適切に変更することを意味する。

## 前提条件
- 対象作業単位のユニット設計生成が完了していること
- NFR実装（実行された場合）が対象作業単位で完了していること
- すべてのユニット設計成果物が利用可能であること
- 作業単位がコード生成可能な状態であること

# パート 1: 計画

## ステップ 1: ユニットコンテキストの分析
- [ ] ユニット設計生成からユニット設計成果物を読み込む
- [ ] ユニットストーリーマップを読み込み、割り当てられたストーリーを把握する
- [ ] ユニットの依存関係とインターフェースを特定する
- [ ] ユニットがコード生成可能な状態であることを確認する

## ステップ 2: 詳細なユニットコード生成計画の作成
- [ ] `aidlc-docs/aidlc-state.md` からワークスペースルートとプロジェクトタイプを読み込む
- [ ] コードの配置場所を決定する（構造パターンはクリティカルルールを参照）
- [ ] **ブラウンフィールドのみ**: リバースエンジニアリングの code-structure.md を確認し、変更する既存ファイルを把握する
- [ ] 正確なパスを記録する（aidlc-docs/ は使用不可）
- [ ] ユニット生成の明示的な手順を作成する:
  - プロジェクト構造のセットアップ（グリーンフィールドのみ）
  - ビジネスロジックの生成
  - ビジネスロジックのユニットテスト
  - ビジネスロジックのサマリー
  - APIレイヤーの生成
  - APIレイヤーのユニットテスト
  - APIレイヤーのサマリー
  - リポジトリレイヤーの生成
  - リポジトリレイヤーのユニットテスト
  - リポジトリレイヤーのサマリー
  - フロントエンドコンポーネントの生成（該当する場合）
  - フロントエンドコンポーネントのユニットテスト（該当する場合）
  - フロントエンドコンポーネントのサマリー（該当する場合）
  - データベースマイグレーションスクリプト（データモデルが存在する場合）
  - ドキュメントの生成（APIドキュメント、READMEの更新）
  - デプロイメント成果物の生成
- [ ] 各手順に連番を付ける
- [ ] ユーザーストーリーへのマッピング参照を含める
- [ ] 各手順にチェックボックス [ ] を追加する

## ステップ 3: ユニット生成コンテキストの記載
- [ ] このユニットについて以下を含める:
  - このユニットで実装されるユーザーストーリー
  - 他のユニット・サービスへの依存関係
  - 期待されるインターフェースとコントラクト
  - このユニットが所有するデータベースエンティティ
  - サービス境界と責務

## ステップ 4: ユニット計画ドキュメントの作成
- [ ] 完全な計画を `aidlc-docs/construction/plans/{unit-name}-code-generation-plan.md` として保存する
- [ ] 手順番号（ステップ 1、ステップ 2 等）を含める
- [ ] ユニットのコンテキストと依存関係を含める
- [ ] ユーザーストーリーのトレーサビリティを含める
- [ ] 計画が手順ごとに実行可能であることを確認する
- [ ] この計画がコード生成の唯一の情報源であることを明記する

## ステップ 5: ユニット計画のサマリー提示
- [ ] ユニットコード生成計画のサマリーをユーザーに提示する
- [ ] ユニット生成のアプローチを強調する
- [ ] 手順の順序とユーザーストーリーのカバレッジを説明する
- [ ] 手順の総数と推定スコープを記載する

## ステップ 6: 承認プロンプトのログ記録
- [ ] 承認を求める前に、タイムスタンプ付きでプロンプトを `aidlc-docs/audit.md` に記録する
- [ ] 完全なユニットコード生成計画への参照を含める
- [ ] ISO 8601タイムスタンプ形式を使用する

## ステップ 7: 明示的な承認の待機
- [ ] ユーザーがユニットコード生成計画を明示的に承認するまで進まない
- [ ] 承認は計画全体と生成シーケンス全体をカバーするものでなければならない
- [ ] ユーザーが変更を要求した場合は、計画を更新して承認プロセスを繰り返す

## ステップ 8: 承認応答の記録
- [ ] ユーザーの承認応答をタイムスタンプ付きで `aidlc-docs/audit.md` に記録する
- [ ] ユーザーの応答テキストをそのまま含める
- [ ] 承認ステータスを明確にマークする

## ステップ 9: 進捗の更新
- [ ] `aidlc-state.md` でコード生成パート 1（計画）を完了としてマークする
- [ ] 「現在のステータス」セクションを更新する
- [ ] コード生成への移行を準備する

# パート 2: 生成

## ステップ 10: ユニットコード生成計画の読み込み
- [ ] `aidlc-docs/construction/plans/{unit-name}-code-generation-plan.md` から完全な計画を読み込む
- [ ] 次の未完了ステップ（最初の [ ] チェックボックス）を特定する
- [ ] そのステップのコンテキスト（ユニット・依存関係・ユーザーストーリー）を読み込む

## ステップ 11: 現在のステップの実行
- [ ] 計画からターゲットディレクトリを確認する（aidlc-docs/ は使用不可）
- [ ] **ブラウンフィールドのみ**: ターゲットファイルが存在するか確認する
- [ ] 現在のステップに記載された内容を正確に生成する:
  - **ファイルが存在する場合**: インプレースで変更する（`ClassName_modified.java` や `ClassName_new.java` 等は絶対に作成しない）
  - **ファイルが存在しない場合**: 新規ファイルを作成する
- [ ] 正しい場所に書き込む:
  - **アプリケーションコード**: プロジェクト構造に従いワークスペースルートへ
  - **ドキュメント**: `aidlc-docs/construction/{unit-name}/code/`（マークダウンのみ）
  - **ビルド・設定ファイル**: ワークスペースルートへ
- [ ] ユニットのユーザーストーリー要件に従う
- [ ] 依存関係とインターフェースを尊重する

## ステップ 12: 進捗の更新
- [ ] ユニットコード生成計画で完了したステップを [x] にマークする
- [ ] 生成が完了したユニットのユーザーストーリーを [x] にマークする
- [ ] `aidlc-docs/aidlc-state.md` の現在のステータスを更新する
- [ ] **ブラウンフィールドのみ**: 重複ファイルが作成されていないことを確認する（例: `ClassName.java` と並存する `ClassName_modified.java` がないこと）
- [ ] 生成されたすべての成果物を保存する

## ステップ 13: 生成の継続または完了
- [ ] 残りのステップがある場合はステップ 10 に戻る
- [ ] すべてのステップが完了した場合は完了メッセージの提示に進む

## ステップ 14: 完了メッセージの提示
- 以下の構成で完了メッセージを提示する:
     1. **完了の告知**（必須）: 常に以下から始める:

```markdown
# 💻 Code Generation Complete - [unit-name]
```

     2. **AIサマリー**（任意）: 箇条書きサマリーを提供する
        - **ブラウンフィールド**: 変更ファイルと新規作成ファイルを区別する（例: "• 変更: `src/services/user-service.ts`"、"• 作成: `src/services/auth-service.ts`"）
        - **グリーンフィールド**: 作成したファイルをパス付きで列挙する（例: "• 作成: `src/services/user-service.ts`"）
        - テスト・ドキュメント・デプロイメント成果物をパス付きで列挙する
        - 事実に基づき、ワークフローの指示は含めない
     3. **フォーマット済みワークフローメッセージ**（必須）: 常に以下の正確な形式で終了する:

```markdown
> **📋 <u>**REVIEW REQUIRED:**</u>**  
> Please examine the generated code at:
> - **Application Code**: `[actual-workspace-path]`
> - **Documentation**: `aidlc-docs/construction/[unit-name]/code/`



> **🚀 <u>**WHAT'S NEXT?**</u>**
>
> **You may:**
>
> 🔧 **Request Changes** - Ask for modifications to the generated code based on your review  
> ✅ **Continue to Next Stage** - Approve code generation and proceed to **[next-unit/Build & Test]**

---
```

## ステップ 15: 明示的な承認の待機
- ユーザーが生成されたコードを明示的に承認するまで進まない
- 承認は明確かつ曖昧さのないものでなければならない
- ユーザーが変更を要求した場合は、コードを更新して承認プロセスを繰り返す

## ステップ 16: 承認の記録と進捗の更新
- タイムスタンプ付きで audit.md に承認を記録する
- ユーザーの承認応答をタイムスタンプ付きで記録する
- aidlc-state.md でこのユニットのコード生成ステージを完了としてマークする

## クリティカルルール

### コード配置ルール
- **アプリケーションコード**: ワークスペースルートのみ（aidlc-docs/ は絶対に使用しない）
- **ドキュメント**: aidlc-docs/ のみ（マークダウンのサマリー）
- **ワークスペースルートの確認**: コード生成前に aidlc-state.md から読み込む

**プロジェクトタイプ別の構造パターン**:
- **ブラウンフィールド**: 既存の構造を使用する（例: `src/main/java/`、`lib/`、`pkg/`）
- **グリーンフィールド・シングルユニット**: ワークスペースルートに `src/`、`tests/`、`config/`
- **グリーンフィールド・マルチユニット（マイクロサービス）**: `{unit-name}/src/`、`{unit-name}/tests/`
- **グリーンフィールド・マルチユニット（モノリス）**: `src/{unit-name}/`、`tests/{unit-name}/`

### ブラウンフィールドのファイル変更ルール
- 生成前にファイルが存在するか確認する
- 存在する場合: インプレースで変更する（`ClassName_modified.java` 等のコピーは絶対に作成しない）
- 存在しない場合: 新規ファイルを作成する
- 生成後に重複ファイルがないことを確認する（ステップ 12）

### 計画フェーズのルール
- すべての生成作業に対して明示的で番号付きの手順を作成する
- 計画にユーザーストーリーのトレーサビリティを含める
- ユニットのコンテキストと依存関係を記録する
- 生成前にユーザーの明示的な承認を得る

### 生成フェーズのルール
- **ハードコードされたロジックは禁止**: ユニット計画に記載された内容のみを実行する
- **計画に厳密に従う**: 手順の順序を変えない
- **チェックボックスの更新**: 各ステップ完了直後に [x] にマークする
- **ユーザーストーリーのトレーサビリティ**: 機能が実装されたらユニットのユーザーストーリーを [x] にマークする
- **依存関係の尊重**: ユニットの依存関係が満たされた場合にのみ実装する

### 自動化対応コードのルール
UI コード（Web・モバイル・デスクトップ）を生成する場合は、要素が自動化に対応していることを確認する:
- インタラクティブな要素（ボタン・入力欄・リンク・フォーム）に `data-testid` 属性を付与する
- 一貫した命名規則を使用する: `{component}-{element-role}`（例: `login-form-submit-button`、`user-list-search-input`）
- レンダリング間で変化する動的またはオートジェネレートされたIDは避ける
- `data-testid` の値は変更を安定させる（要素の目的が変わった場合にのみ変更する）

## 完了基準
- 完全なユニットコード生成計画が作成・承認されていること
- ユニットコード生成計画のすべての手順が [x] にマークされていること
- 計画に従いすべてのユニットのユーザーストーリーが実装されていること
- すべてのコードとテストが生成されていること（テストの実行はビルドとテストフェーズで行う）
- デプロイメント成果物が生成されていること
- 完全なユニットがビルドと検証の準備が整っていること
