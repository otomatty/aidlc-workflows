# アプリケーション設計 - 詳細手順

## 目的
**高レベルのコンポーネント特定とサービス層設計**

アプリケーション設計が対象とする領域:
- 主要な機能コンポーネントとその責務の特定
- コンポーネントインターフェースの定義（詳細なビジネスロジックは除く）
- オーケストレーション用サービス層の設計
- コンポーネント間の依存関係とコミュニケーションパターンの確立

**注意**: 詳細なビジネスロジックの設計は、後続のFunctional Design（作業単位(unit of work)ごと、CONSTRUCTIONフェーズ）で行います。

## 前提条件
- Workspace Detectionが完了していること
- Requirements Analysis（要件分析）を推奨（機能的コンテキストを提供する）
- User Stories（ユーザーストーリー）を推奨（設計判断の指針となる）
- 実行計画においてApplication Designステージを実行するよう指示されていること

## ステップバイステップ実行

### 1. コンテキストの分析
- `aidlc-docs/inception/requirements/requirements.md` および `aidlc-docs/inception/user-stories/stories.md` を読む
- 主要なビジネス機能と機能領域を特定する
- 設計のスコープと複雑さを判断する

### 2. アプリケーション設計計画の作成
- アプリケーション設計用にチェックボックス [] 付きの計画を生成する
- コンポーネント、責務、メソッド、ビジネスルール、サービスに焦点を当てる
- 各ステップとサブステップにチェックボックス [] を付ける

### 3. 必須設計成果物の計画への組み込み
- 以下の必須成果物を設計計画に**必ず**含めること:
  - [ ] components.md を生成する（コンポーネント定義と高レベルの責務）
  - [ ] component-methods.md を生成する（メソッドシグネチャ。ビジネスルールの詳細はFunctional Designで後述）
  - [ ] services.md を生成する（サービス定義とオーケストレーションパターン）
  - [ ] component-dependency.md を生成する（依存関係とコミュニケーションパターン）
  - [ ] 設計の完全性と整合性を検証する

### 4. コンテキストに適した質問の生成
**指示**: 要件とストーリーを分析し、この特定のアプリケーション設計に関連する質問を生成する。以下のカテゴリを指針として使用すること。各カテゴリを評価し、適用可能かどうか疑問がある場合は、スキップせずに質問する — 過信は悪い結果につながる（overconfidence-prevention.md 参照）。

- [Answer]: タグ形式を使って質問を埋め込む
- 曖昧な点、不足している情報、明確化が必要な領域に焦点を当てる
- 利用者の入力によって設計判断が改善される箇所には質問を生成する
- **疑問がある場合は質問する** — 過信は設計の品質低下につながる

**評価すべき質問カテゴリ**（すべてのカテゴリを検討すること）:
- **コンポーネント特定** — コンポーネントの境界、整理、グルーピング戦略について質問する
- **コンポーネントメソッド** — メソッドシグネチャ、入出力の期待値、インターフェース契約について質問する（詳細なビジネスルールは後述）
- **サービス層設計** — サービスオーケストレーション、境界、調整パターンについて質問する
- **コンポーネント依存関係** — コミュニケーションパターン、依存関係の管理、結合の懸念について質問する
- **設計パターン** — アーキテクチャスタイルの好み、パターンの選択、設計上の制約について質問する

### 5. アプリケーション設計計画の保存
- `aidlc-docs/inception/plans/application-design-plan.md` として保存する
- ユーザー入力用のすべての [Answer]: タグを含める
- 計画がすべての設計側面を網羅していることを確認する

### 6. ユーザー入力の要求
- 計画ドキュメント内の [Answer]: タグを直接記入するよう求める
- 設計判断の重要性を強調する
- [Answer]: タグの記入方法を明確に説明する

### 7. 回答の収集
- [Answer]: タグを使ってすべての質問に回答するのを待つ
- すべての [Answer]: タグが記入されるまで次に進まない
- ドキュメントを確認し、未記入の [Answer]: タグがないことを確認する

### 8. 回答の分析（必須）
次のステップに進む前に、すべてのユーザー回答を以下の観点で注意深くレビューすること:
- **曖昧または不明確な回答**: 「〜の組み合わせ」「〜の中間」「よくわからない」「場合による」
- **定義されていない基準や用語**: 明確な定義のない概念への言及
- **矛盾する回答**: 互いに矛盾する回答
- **設計の詳細が不足している回答**: 具体的なガイダンスが不足している回答
- **複数の選択肢を組み合わせた回答**: 明確な判断基準なしに異なるアプローチを組み合わせた回答

### 9. 必須フォローアップ質問
ステップ8の分析で曖昧な回答が見つかった場合は、次のことを**必ず**行うこと:
- [Answer]: タグを使って具体的なフォローアップ質問を計画ドキュメントに追加する
- すべての曖昧さが解消されるまで承認ステップに進まない
- 必要なフォローアップの例:
  - 「『AとBの組み合わせ』とおっしゃっていましたが、AとBのどちらを使うかを決定する具体的な基準を教えてください」
  - 「『AとBの中間』とおっしゃっていましたが、その具体的な中間のアプローチを定義できますか？」
  - 「『よくわからない』とおっしゃっていましたが、決定するためにどのような追加情報が必要ですか？」
  - 「『複雑さによる』とおっしゃっていましたが、複雑さのレベルはどのように定義しますか？」

### 10. アプリケーション設計成果物の生成
- 承認済みの計画を実行して設計成果物を生成する
- `aidlc-docs/inception/application-design/components.md` を作成する（内容）:
  - コンポーネント名と目的
  - コンポーネントの責務
  - コンポーネントインターフェース
- `aidlc-docs/inception/application-design/component-methods.md` を作成する（内容）:
  - 各コンポーネントのメソッドシグネチャ
  - 各メソッドの高レベルな目的
  - 入出力の型
  - 注意: 詳細なビジネスルールはFunctional Design（作業単位(unit of work)ごと、CONSTRUCTIONフェーズ）で定義される
- `aidlc-docs/inception/application-design/services.md` を作成する（内容）:
  - サービス定義
  - サービスの責務
  - サービスの相互作用とオーケストレーション
- `aidlc-docs/inception/application-design/component-dependency.md` を作成する（内容）:
  - 関係を示す依存関係マトリクス
  - コンポーネント間のコミュニケーションパターン
  - データフロー図
- 上記の複数の設計ドキュメントを一つにまとめた `aidlc-docs/inception/application-design/application-design.md` を作成する

### 11. 承認のログ記録
- タイムスタンプ付きの承認プロンプトを `aidlc-docs/audit.md` に記録する
- 完全な承認プロンプトのテキストを含める
- ISO 8601形式のタイムスタンプを使用する

### 12. 完了メッセージの提示

```markdown
# 🏗️ Application Design Complete

[作成されたアプリケーション設計成果物のAI生成サマリー（箇条書き）]

> **📋 <u>**REVIEW REQUIRED:**</u>**  
> Please examine the application design artifacts at: `aidlc-docs/inception/application-design/`

> **🚀 <u>**WHAT'S NEXT?**</u>**
>
> **You may:**
>
> 🔧 **Request Changes** - Ask for modifications to the application design if required
> [IF Units Generation is skipped:]
> 📝 **Add Units Generation** - Choose to include **Units Generation** stage (currently skipped)
> ✅ **Approve & Continue** - Approve design and proceed to **[Units Generation/CONSTRUCTION PHASE]**
```

### 13. 明示的な承認の待機
- ユーザーがアプリケーション設計を明示的に承認するまで次に進まない
- 承認は明確かつ曖昧さのないものでなければならない
- 変更を要求された場合は設計を更新し、承認プロセスを繰り返す

### 14. 承認応答の記録
- ユーザーの承認応答をタイムスタンプ付きで `aidlc-docs/audit.md` に記録する
- 正確なユーザー応答テキストを含める
- 承認ステータスを明確にマークする

### 15. 進捗の更新
- `aidlc-docs/aidlc-state.md` のApplication Designステージを完了としてマークする
- 「現在のステータス」セクションを更新する
- 次のステージへの移行を準備する
