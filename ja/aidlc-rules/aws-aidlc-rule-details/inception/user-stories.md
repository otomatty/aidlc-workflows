# ユーザーストーリー - 詳細手順

## 目的
**要件をユーザー中心のストーリーと受け入れ基準に変換する**

ユーザーストーリーでは以下を行う：
- ビジネス要件をユーザー中心のナラティブに変換する
- 各ストーリーの明確な受け入れ基準を定義する
- さまざまなステークホルダータイプを表すユーザーペルソナを作成する
- チーム全体の共通理解を確立する
- 実装のためのテスト可能な仕様を提供する

## 前提条件
- ワークスペース検出が完了していること
- 要件分析が推奨される（利用可能であれば要件を参照できる）
- ワークフロープランニングでユーザーストーリーステージを実行することが示されていること

## インテリジェントな評価ガイドライン

**ユーザーストーリーを実行するタイミング**: 進める前にこの拡張評価を行うこと：

### 高優先度（常に実行）
- **新しいユーザー機能**: ユーザーが直接操作する新しい機能
- **ユーザー体験の変更**: 既存のユーザーワークフローやインターフェースへの変更
- **マルチペルソナシステム**: 異なるタイプのユーザーに対応するアプリケーション
- **顧客向け API**: 外部ユーザーやシステムが使用するサービス
- **複雑なビジネスロジック**: 複数のシナリオやビジネスルールを持つ要件
- **クロスチームプロジェクト**: 複数のチームにまたがる共通理解が必要な作業

### 中優先度（複雑性を評価）
- **バックエンドのユーザー影響**: ユーザー体験に間接的に影響する内部変更
- **パフォーマンスの改善**: ユーザーに見えるメリットのある機能強化
- **統合作業**: ユーザーワークフローに影響するシステムの接続
- **データの変更**: ユーザーデータ、レポート、分析に影響する変更
- **セキュリティの強化**: ユーザー認証や権限に影響する変更

### 複雑性評価の要素
中優先度の場合、以下のいずれかに該当する場合はユーザーストーリーを実行する：
- **スコープ**: 変更が複数のコンポーネントやユーザータッチポイントにまたがる
- **曖昧さ**: 要件にストーリーで明確化できる不明な点がある
- **リスク**: ビジネスへの影響が大きい、または誤解の可能性がある
- **ステークホルダー**: 複数のビジネスステークホルダーが要件に関与している
- **テスト**: ユーザー受け入れテストが必要
- **選択肢**: 複数の有効な実装アプローチが存在する

### シンプルなケースのみスキップ
- **純粋なリファクタリング**: ユーザーへの影響がゼロの内部コード改善
- **局所的なバグ修正**: スコープが明確な単純で明確な修正
- **インフラストラクチャのみ**: ユーザー向けの影響がない変更
- **開発者ツール**: ビルドプロセス、CI/CD、または開発環境の変更
- **ドキュメント**: 機能に影響しない更新

### デフォルトの判断ルール
**迷ったときは、ユーザーストーリーを含め、かつ明確化の質問を行う。** 適切な質問を含む包括的なストーリーを作成するオーバーヘッドは、通常、以下のメリットを下回る：
- 要件の明確な理解
- チームの整合性向上
- テスト基準の改善
- ステークホルダーとのコミュニケーション強化
- 実装リスクの低減
- 開発中のコストのかかる変更の削減
- ユーザー体験の向上

# パート1: 計画

## ステップ1: ユーザーストーリーの必要性を検証する（必須）

**重要**: ユーザーストーリーを進める前に、この評価を行うこと：

### 評価プロセス
1. **リクエストコンテキストの分析**：
   - 元のユーザーリクエストと要件を確認する
   - ユーザー向け変更と内部のみの変更を特定する
   - 作業の複雑性とスコープを評価する
   - ビジネスステークホルダーの関与を評価する

2. **評価基準の適用**：
   - 高優先度の指標に対して確認する（常に実行）
   - 中優先度の要素を評価する（複雑性に基づく判断）
   - スキップすべき単純なケースでないことを確認する

3. **評価の判断を文書化する**：
   - `aidlc-docs/inception/plans/user-stories-assessment.md` を作成する
   - このリクエストにユーザーストーリーが価値をもたらす理由を含める
   - 適用される具体的な評価基準を参照する
   - 期待されるメリット（明確性、テスト、ステークホルダーの整合）を説明する

4. **正当化される場合のみ進める**：
   - ユーザーストーリーはプロジェクトに明確な価値を追加しなければならない
   - 評価では、コンクリートなメリットがオーバーヘッドを上回ることを示す必要がある
   - 判断はプロジェクトのステークホルダーに対して説明可能でなければならない

### 評価ドキュメントのテンプレート
```markdown
# User Stories Assessment

## Request Analysis
- **Original Request**: [Brief summary]
- **User Impact**: [Direct/Indirect/None]
- **Complexity Level**: [Simple/Medium/Complex]
- **Stakeholders**: [List involved parties]

## Assessment Criteria Met
- [ ] High Priority: [List applicable criteria]
- [ ] Medium Priority: [List applicable criteria with complexity justification]
- [ ] Benefits: [Expected value from user stories]

## Decision
**Execute User Stories**: [Yes/No]
**Reasoning**: [Detailed justification]

## Expected Outcomes
- [List specific benefits user stories will provide]
- [How stories will improve project success]
```

## ステップ2: ストーリー計画の作成
- プロダクトオーナーの役割を担う
- ストーリー開発のためのステップバイステップの実行チェックリストを含む包括的な計画を生成する
- 各ステップとサブステップにチェックボックス [] を付ける
- 要件をユーザーストーリーに変換するための方法論とアプローチに焦点を当てる

## ステップ3: コンテキストに適した質問の生成
**指示**: 要件とコンテキストを徹底的に分析し、ストーリーの品質とチームの理解を向上させるための明確化が必要なすべての領域を特定すること。包括的なユーザーストーリー開発を確保するため、積極的に質問すること。

**重要**: ストーリーの品質に影響する可能性のある曖昧さや不足情報が少しでもあれば、質問することをデフォルトとする。不完全または不明確なストーリーを作成するよりも、多すぎるくらい質問する方が望ましい。

**質問フォーマットのルールについては `common/question-format-guide.md` を参照**

- [Answer]: タグ形式を使用して質問を埋め込む
- あいまいさ、不足情報、明確化が必要な領域に焦点を当てる
- ユーザーの入力がストーリー作成判断を改善するあらゆる箇所で質問を生成する
- **迷ったときは質問する** — 過信はストーリーの質低下につながる

**評価すべき質問カテゴリ**（全カテゴリを検討すること）：
- **ユーザーペルソナ** — ユーザーの種類、役割、特性、動機について質問する
- **ストーリーの粒度** — 適切な詳細レベル、ストーリーのサイズ、分解アプローチについて質問する
- **ストーリーの形式** — 形式の好み、テンプレートの使用、ドキュメント標準について質問する
- **分解アプローチ** — 構成方法、優先順位付け、グループ化戦略について質問する
- **受け入れ基準** — 詳細レベル、形式、テストアプローチ、検証方法について質問する
- **ユーザージャーニー** — ユーザーワークフロー、インタラクションパターン、体験フローについて質問する
- **ビジネスコンテキスト** — ビジネス目標、成功指標、ステークホルダーのニーズについて質問する
- **技術的制約** — 技術的な制限、統合要件、システム境界について質問する

## ステップ4: 必須ストーリー成果物を計画に含める
- 以下の必須成果物をストーリー計画に**必ず**含めること：
  - [ ] INVEST 基準に従ったユーザーストーリーを含む stories.md を生成する
  - [ ] ユーザーアーキタイプと特性を含む personas.md を生成する
  - [ ] ストーリーが独立していること(Independent)、交渉可能であること(Negotiable)、価値があること(Valuable)、見積もり可能であること(Estimable)、小さいこと(Small)、テスト可能であること(Testable)を確認する
  - [ ] 各ストーリーの受け入れ基準を含める
  - [ ] ペルソナを関連するユーザーストーリーにマッピングする

## ステップ5: ストーリーのオプションを提示する
- 計画ドキュメントにストーリー分解のさまざまなアプローチを含める：
  - **ユーザージャーニー基準**: ストーリーはユーザーワークフローとインタラクションに従う
  - **機能基準**: ストーリーはシステムの機能とケイパビリティを中心に構成される
  - **ペルソナ基準**: ストーリーは異なるユーザータイプとそのニーズごとにグループ化される
  - **ドメイン基準**: ストーリーはビジネスドメインまたはコンテキストを中心に構成される
  - **エピック基準**: ストーリーはサブストーリーを持つ階層的なエピックとして構成される
- 各アプローチのトレードオフとメリットを説明する
- 明確な判断基準を持つハイブリッドアプローチを許容する

## ステップ6: ストーリー計画の保存
- 埋め込まれた質問を含む完全なストーリー計画を `aidlc-docs/inception/plans/` ディレクトリに保存する
- ファイル名: `story-generation-plan.md`
- ユーザー入力用の [Answer]: タグをすべて含める
- 計画が包括的で、ストーリー開発のすべての側面をカバーしていることを確認する

## ステップ7: ユーザー入力の要請
- ストーリー計画ドキュメント内のすべての [Answer]: タグを直接記入するようユーザーに依頼する
- 監査証跡と判断文書の重要性を強調する
- [Answer]: タグへの記入方法について明確な指示を提供する
- 進める前にすべての質問に回答が必要であることを説明する

## ステップ8: 回答の収集
- ドキュメント内の [Answer]: タグを使ってすべての質問に回答するまで待つ
- すべての [Answer]: タグが記入されるまで進めない
- ドキュメントを確認し、空白の [Answer]: タグが残っていないことを確認する

## ステップ9: 回答の分析（必須）
進める前に、すべてのユーザー回答を以下の観点から慎重に確認すること：
- **曖昧または不明確な回答**: 「AとBの混在」「どちらかといえば」「わからない」「場合による」「たぶん」「おそらく」
- **未定義の基準や用語**: 明確な定義のない概念への言及
- **矛盾した回答**: 互いに相反する回答
- **生成の詳細が不足している回答**: 実装のための具体的なガイダンスを欠く回答
- **複数の選択肢を組み合わせた回答**: 明確な判断基準なしに異なるアプローチを統合した回答
- **不完全な説明**: 定義されていない外部要因を参照する回答
- **前提に基づく回答**: 明示されていない知識を前提とする回答

## ステップ10: 必須のフォローアップ質問
ステップ9の分析で曖昧な回答が見つかった場合、必ず以下を実行すること：
- [Answer]: タグを使って別の質問ファイルを作成する
- すべての曖昧さが完全に解消されるまで承認プロセスに進まない
- **重要**: 徹底的に行うこと — 不明確な回答すべてにフォローアップ質問をすること
- 必要なフォローアップの例：
  - 「『AとBの混在』と述べましたが、AとBのどちらを使うかを決める具体的な基準は何ですか？」
  - 「『AとBの間』と述べましたが、具体的な中間アプローチを定義できますか？」
  - 「『わからない』と述べましたが、判断に必要な追加情報は何ですか？」
  - 「『複雑さによる』と述べましたが、複雑さのレベルと閾値をどう定義しますか？」
  - 「『ハイブリッドアプローチ』を選びましたが、各方法をいつ使うかの具体的なルールは何ですか？」
  - 「『おそらくX』と述べましたが、確実にXとなる要素と確実にXでなくなる要素は何ですか？」
  - 「『標準的な手法』と述べましたが、その標準的な手法を具体的に定義できますか？」

## ステップ11: 実装の詳細を避ける
- 優先順位付けや開発タスクではなく、ストーリー作成の方法論に焦点を当てる
- この段階では技術的な生成について議論しない
- 開発タイムラインやスプリント計画を作成しない
- ストーリーの構造と形式の判断に焦点を当てる

## ステップ12: 承認プロンプトのログ記録
- 承認を求める前に、`aidlc-docs/audit.md` にタイムスタンプ付きでプロンプトを記録する
- 承認プロンプトの完全なテキストを含める
- ISO 8601 タイムスタンプ形式を使用する

## ステップ13: 計画の明示的な承認を待つ
- ユーザーがストーリーのアプローチを明示的に承認するまで進めない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合は、計画を更新して承認プロセスを繰り返す

## ステップ14: 承認応答の記録
- `aidlc-docs/audit.md` にタイムスタンプ付きでユーザーの承認応答を記録する
- ユーザーの正確な応答テキストを含める
- 承認状態を明確にマークする

# 重要なルール

## 計画フェーズのルール
- **コンテキストに適した質問**: このコンテキストに関連する質問のみを行う
- **必須の回答分析**: 進める前に常に回答を曖昧さがないか分析する
- **曖昧さがある状態で進めない**: 生成前にすべての曖昧な回答を解消しなければならない
- **明示的な承認が必要**: 生成を開始する前にユーザーが計画を承認しなければならない

## 生成フェーズのルール
- **ハードコードされたロジックは禁止**: ストーリー生成計画に記述された内容のみを実行する
- **計画に正確に従う**: ステップの順序から逸脱しない
- **チェックボックスを更新する**: 各ステップ完了直後に [x] としてマークする
- **承認された方法論を使用する**: 計画フェーズからのストーリーアプローチに従う
- **完了を確認する**: 進める前にすべてのストーリー成果物が完成していることを確認する

## 完了基準
- すべての計画の質問が回答され、曖昧さが解消されている
- ストーリー計画がユーザーによって明示的に承認されている
- ストーリー生成計画のすべてのステップが [x] としてマークされている
- 計画に従ってすべてのストーリー成果物が生成されている（stories.md、personas.md）
- 生成されたストーリーがユーザーによって明示的に承認されている
- ストーリーが検証され、次のステージに向けて準備完了
