# 要件分析（アダプティブ）

**役割**: プロダクトオーナーとして行動すること

**アダプティブフェーズ**: 常に実行される。詳細レベルは問題の複雑性に応じて適応する。

**アダプティブ深度の説明については [depth-levels.md](../common/depth-levels.md) を参照**

## 前提条件
- ワークスペース検出が完了していること
- リバースエンジニアリングが完了していること（ブラウンフィールドの場合）

## 実行手順

### ステップ1: リバースエンジニアリングのコンテキストを読み込む（利用可能な場合）

**ブラウンフィールドプロジェクトの場合**：
- `aidlc-docs/inception/reverse-engineering/architecture.md` を読み込む
- `aidlc-docs/inception/reverse-engineering/component-inventory.md` を読み込む
- `aidlc-docs/inception/reverse-engineering/technology-stack.md` を読み込む
- これらを使用して、リクエストを分析する際に既存システムを理解する

### ステップ2: ユーザーリクエストの分析（インテント分析）

#### 2.1 リクエストの明確性
- **明確**: 具体的、明確に定義されており、実行可能
- **曖昧**: 一般的、不明確、明確化が必要
- **不完全**: 重要な情報が欠けている

#### 2.2 リクエストの種類
- **新機能**: 新しい機能の追加
- **バグ修正**: 既存の問題の修正
- **リファクタリング**: コード構造の改善
- **アップグレード**: 依存関係やフレームワークの更新
- **移行**: 異なる技術への移行
- **機能強化**: 既存機能の改善
- **新規プロジェクト**: ゼロからの開始

#### 2.3 初期スコープ見積もり
- **単一ファイル**: 1つのファイルへの変更
- **単一コンポーネント**: 1つのコンポーネント/パッケージへの変更
- **複数コンポーネント**: 複数のコンポーネントにまたがる変更
- **システム全体**: システム全体に影響する変更
- **クロスシステム**: 複数のシステムに影響する変更

#### 2.4 初期複雑性見積もり
- **自明**: 単純で直接的な変更
- **シンプル**: 明確な実装パス
- **中程度**: ある程度の複雑性、複数の考慮事項
- **複雑**: 大きな複雑性、多くの考慮事項

### ステップ3: 要件の深度を決定する

**リクエスト分析に基づき、深度を決定する：**

**最小深度** — 以下の場合に使用：
- リクエストが明確かつシンプル
- 詳細な要件が不要
- 基本的な理解を文書化するだけでよい

**標準深度** — 以下の場合に使用：
- リクエストに明確化が必要
- 機能要件と非機能要件が必要
- 通常の複雑性

**包括的深度** — 以下の場合に使用：
- 複数のステークホルダーを持つ複雑なプロジェクト
- 高リスクまたはクリティカルなシステム
- トレーサビリティを伴う詳細な要件が必要

### ステップ4: 現在の要件を評価する

ユーザーが提供したものを分析する：
   - インテントの記述や説明（audit.md にすでにログ済み）
   - 既存の要件ドキュメント（言及されていればワークスペースを検索）
   - 貼り付けられたコンテンツやファイル参照
   - Markdown 以外のドキュメントは Markdown 形式に変換する

### ステップ5: 徹底的な完全性分析

**重要**: 要件の完全性を評価するために包括的な分析を行うこと。曖昧さや不足情報が少しでもあれば、質問することをデフォルトとする。

**必須**: 以下のすべての領域を評価し、不明な点があれば質問すること：
- **機能要件**: 中核機能、ユーザーインタラクション、システムの振る舞い
- **非機能要件**: パフォーマンス、セキュリティ、スケーラビリティ、ユーザビリティ
- **ユーザーシナリオ**: ユースケース、ユーザージャーニー、エッジケース、エラーシナリオ
- **ビジネスコンテキスト**: 目標、制約、成功基準、ステークホルダーのニーズ
- **技術環境**: 統合ポイント、データ要件、システム境界
- **品質属性**: 信頼性、保守性、テスト可能性、アクセシビリティ

**迷ったときは質問する** — 不完全な要件は、不適切な実装につながる。

### ステップ5.1: 拡張機能のオプトインプロンプト

**必須**: 読み込まれたすべての `*.opt-in.md` ファイル（ワークフロー開始時に `extensions/` サブディレクトリから読み込まれる）をスキャンして、`## Opt-In Prompt` セクションを探す。そのセクションを宣言している拡張機能ごとに、ステップ6で作成する質問ファイルにそのオプトイン質問を含める。各オプトイン質問は、ユーザーとの会話と同じ言語で提示する。

回答を受け取った後：
1. 各拡張機能の有効化状態を `aidlc-docs/aidlc-state.md` の `## Extension Configuration` の下に記録する：

```markdown
## Extension Configuration
| Extension | Enabled | Decided At |
|---|---|---|
| [Extension Name] | [Yes/No] | Requirements Analysis |
```

2. **遅延ルール読み込み**: ユーザーがオプトインした拡張機能ごとに、今すぐ完全なルールファイルを読み込む。ルールファイルは命名規則で決まる：オプトインファイル名から `.opt-in.md` を除いて `.md` を付加する（例: `security-baseline.opt-in.md` → `security-baseline.md`）。ユーザーがオプトアウトした拡張機能については、完全なルールファイルを読み込まないこと。

### ステップ6: 確認質問の生成（プロアクティブなアプローチ）
   - 要件が例外的に明確かつ完全でない限り、**常に** `aidlc-docs/inception/requirements/requirement-verification-questions.md` を作成する
   - 不足・不明・曖昧な領域について質問する
   - 機能要件、非機能要件、ユーザーシナリオ、ビジネスコンテキストに焦点を当てる
   - 質問ドキュメント内のすべての [Answer]: タグを直接記入するようユーザーに依頼する
   - 回答の選択肢を複数提示する場合：
     - 選択肢に A、B、C、D などのラベルを付ける
     - 選択肢が相互排他的で重複しないようにする
     - カスタム回答のオプションを常に含める：「X) その他（[Answer]: タグの後に説明を記入してください）」
   - ドキュメントへのユーザー回答を待つ
   - **必須**: すべての回答を曖昧さがないか分析し、必要に応じてフォローアップ質問を作成する
   - **必須**: すべての曖昧さが解消されるか、ユーザーが明示的に進めるよう依頼するまで質問を続ける

### ⛔ ゲート: ユーザーの回答を待つ
requirement-verification-questions.md のすべての質問が回答・検証されるまでステップ7に進まないこと。
質問ファイルをユーザーに提示して停止する。

### ステップ7: 要件ドキュメントの生成
   - **前提条件**: ステップ6のゲートを通過すること — すべての回答を受け取り、分析済みであること
   - `aidlc-docs/inception/requirements/requirements.md` を作成する
   - 先頭にインテント分析サマリーを含める：
     - ユーザーリクエスト
     - リクエストの種類
     - スコープ見積もり
     - 複雑性見積もり
   - 機能要件と非機能要件の両方を含める
   - 確認質問へのユーザーの回答を組み込む
   - 主要要件の簡潔なサマリーを提供する

### ステップ8: 状態追跡の更新

`aidlc-docs/aidlc-state.md` を更新する：

```markdown
## Stage Progress
### 🔵 INCEPTION PHASE
- [x] Workspace Detection
- [x] Reverse Engineering (if applicable)
- [x] Requirements Analysis
```

### ステップ9: ログ記録と進行
   - `aidlc-docs/audit.md` にタイムスタンプ付きで承認プロンプトを記録する
   - 完了メッセージを以下の構成で提示する：
     1. **完了アナウンス**（必須）: 常にこれで始める：

```markdown
# 🔍 Requirements Analysis Complete
```

     2. **AI サマリー**（任意）: 要件の構造化された箇条書きサマリーを提供する
        - 形式：「Requirements analysis has identified [project type/complexity]:」
        - 主要な機能要件を列挙（箇条書き）
        - 主要な非機能要件を列挙（箇条書き）
        - 関連するアーキテクチャの考慮事項や技術的判断があれば言及する
        - ワークフロー指示は含めない（「please review」「let me know」「proceed to next phase」「before we proceed」など）
        - 事実に基づき、コンテンツに集中する
     3. **フォーマット済みワークフローメッセージ**（必須）: 常にこの正確な形式で終わる：

```markdown
> **📋 <u>**REVIEW REQUIRED:**</u>**  
> Please examine the requirements document at: `aidlc-docs/inception/requirements/requirements.md`



> **🚀 <u>**WHAT'S NEXT?**</u>**
>
> **You may:**
>
> 🔧 **Request Changes** -  Ask for modifications to the requirements if required based on your review 
> [IF User Stories will be skipped, add this option:]
> 📝 **Add User Stories** - Choose to Include **User Stories** stage (currently skipped based on project simplicity)  
> ✅ **Approve & Continue** - Approve requirements and proceed to **[User Stories/Workflow Planning]**

---
```

**注意**: 「Add User Stories」オプションは、ユーザーストーリーステージをスキップする場合のみ含める。[User Stories/Workflow Planning] は実際の次のステージ名に置き換えること。

   - 進める前に明示的なユーザー承認を待つ
   - タイムスタンプ付きで承認応答を記録する
   - aidlc-state.md の要件分析ステージを完了としてマークする
