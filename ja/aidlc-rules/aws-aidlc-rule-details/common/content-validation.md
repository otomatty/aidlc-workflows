# コンテンツ検証ルール

## 必須：ファイル作成前のコンテンツ検証

**重要**: 生成したコンテンツはすべて、ファイルへの書き込み前に検証を行い、パースエラーを防ぐ必要があります。

## ASCII図の標準規格

**重要**: ASCII図を含むファイルを作成する前に、必ず以下を行ってください：

1. **ロード** `common/ascii-diagram-standards.md`
2. **各図を検証** する：
   - 1行あたりの文字数を数える（すべての行が同じ幅でなければならない）
   - 使用可能な文字：`+` `-` `|` `^` `v` `<` `>` およびスペースのみ
   - Unicodeの罫線文字は使用禁止
   - スペースのみを使用（タブは不可）
3. **アライメントをテスト** してボックスの角が縦に揃っているか確認する

**パターンと検証チェックリストは `common/ascii-diagram-standards.md` を参照してください。**

## Mermaid図の検証

### 必須の検証手順
1. **構文チェック**：ファイル作成前にMermaidの構文を検証する
2. **特殊文字のエスケープ**：特殊文字が適切にエスケープされていることを確認する
3. **フォールバックコンテンツ**：Mermaidの検証が失敗した場合のテキスト代替を提供する

### Mermaid検証ルール
```markdown
## Mermaid図を含むファイルを作成する前に：

1. ノードIDに無効な文字が含まれていないか確認する（英数字とアンダースコアのみ使用可）
2. ラベル内の特殊文字をエスケープする：" → \" および ' → \'
3. フローチャートの構文を検証する：ノード接続が有効であること
4. シンプルな検証で図のパースをテストする

## フォールバック：Mermaidの検証が失敗した場合はテキストベースのワークフロー表現を使用する
```

### 実装パターン
```markdown
## ワークフローの可視化

### Mermaid図（構文が有効な場合）
```mermaid
[検証済みの図のコンテンツ]
```

### テキスト代替（常に含める）
```
Phase 1: INCEPTION
- Stage 1: Workspace Detection (COMPLETED)
- Stage 2: Requirements Analysis (COMPLETED)
[テキスト表現を続ける]
```

## 一般的なコンテンツ検証

### 作成前の検証チェックリスト
- [ ] 埋め込みコードブロック（Mermaid、JSON、YAML）を検証する
- [ ] 特殊文字のエスケープを確認する
- [ ] Markdown構文の正確さを検証する
- [ ] コンテンツのパース互換性をテストする
- [ ] 複雑な要素にはフォールバックコンテンツを含める

### エラー防止ルール
1. **ツール/コマンドでファイルを書き込む前に必ず検証する**：未検証のコンテンツは書き込まない
2. **特殊文字をエスケープする**：特に図やコードブロック内で
3. **代替手段を提供する**：視覚的コンテンツのテキスト版を含める
4. **構文をテストする**：複雑なコンテンツ構造を検証する

## 検証失敗時の対応

### 検証が失敗した場合
1. **エラーを記録する**：何が検証に失敗したかを記録する
2. **フォールバックコンテンツを使用する**：テキストベースの代替に切り替える
3. **ワークフローを継続する**：コンテンツ検証の失敗でブロックしない
4. **ユーザーに通知する**：パースの制約により簡略化されたコンテンツを使用した旨を伝える
