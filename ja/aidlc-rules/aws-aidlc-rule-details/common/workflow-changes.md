# ワークフロー途中の変更とステージ管理

## 概要

ユーザーはワークフロー(workflow)の途中で実行計画またはステージ実行の変更を要求することがある。このドキュメントは、これらのリクエストを安全かつ効果的に処理するためのガイダンスを提供する。

---

## ワークフロー途中の変更の種類

### 1. スキップしたステージの追加

**シナリオ**：ユーザーが最初にスキップしたステージを追加したい

**例**：「やっぱり、スキップしたけれどユーザーストーリーを追加したい」

**対応方法**：
1. **リクエストを確認する**：「User Storiesステージを追加したいということですね。これによりユーザーストーリーとペルソナが作成されます。よろしいですか？」
2. **依存関係を確認する**：すべての前提条件ステージが完了しているかを確認する
3. **実行計画を更新する**：`execution-plan.md` に根拠とともにステージを追加する
4. **状態を更新する**：`aidlc-state.md` でステージを「PENDING」としてマークする
5. **ステージを実行する**：通常のステージ実行プロセスに従う
6. **変更を記録する**：タイムスタンプと理由とともに `audit.md` に文書化する

**考慮事項**：
- 新しい成果物から恩恵を受けられる後のステージの更新が必要な場合がある
- 新しい情報を組み込むために既存の成果物の修正が必要な場合がある
- タイムラインが延長される

---

### 2. 計画されたステージのスキップ

**シナリオ**：ユーザーが実行予定だったステージをスキップしたい

**例**：「NFR Designステージは今はスキップしましょう」

**対応方法**：
1. **リクエストを確認する**：「NFR Designをスキップするということですね。これはNFRパターンや論理コンポーネントが組み込まれないことを意味します。よろしいですか？」
2. **影響について警告する**：何が欠落するか、潜在的な結果を説明する
3. **明示的な確認を得る**：ユーザーは影響を理解していることを明示的に確認しなければならない
4. **実行計画を更新する**：ステージを「SKIPPED」と理由とともにマークする
5. **状態を更新する**：`aidlc-state.md` でステージを「SKIPPED」としてマークする
6. **後のステージを調整する**：後のステージで手動セットアップが必要になる可能性があることに注意する
7. **変更を記録する**：タイムスタンプと理由とともに `audit.md` に文書化する

**考慮事項**：
- 後のステージが失敗するか手動の介入が必要になる場合がある
- ユーザーは欠落した成果物に対する責任を受け入れる
- 必要であれば後から追加できる

---

### 3. 現在のステージの再開

**シナリオ**：ユーザーが現在のステージの結果に不満でやり直したい

**例**：「このユーザーストーリーは気に入らない。最初からやり直せる？」

**対応方法**：
1. **懸念を理解する**：「ストーリーのどの点を具体的に変更したいですか？」
2. **選択肢を提示する**：
   - **オプションA**：既存の成果物を修正する（より速く、一部の作業を保存できる）
   - **オプションB**：完全な再開（白紙から始める、より時間がかかる）
3. **再開が選択された場合**：
   - 既存の成果物をアーカイブする：`{artifact}.backup.{timestamp}`
   - 計画ファイルのステージチェックボックスをリセットする
   - `aidlc-state.md` でステージを「IN PROGRESS」としてマークする
   - ステージの完了状態をクリアする
   - 最初から再実行する
4. **変更を記録する**：再開の理由と何が変わるかを文書化する

**考慮事項**：
- 既存の作業は失われる（ただしバックアップされる）
- 依存するステージを再度行う必要がある場合がある
- タイムラインが延長される

---

### 4. 前のステージの再開

**シナリオ**：ユーザーが戻って完了したステージをやり直したい

**例**：「以前に行ったアーキテクチャの決定を変更したい」

**対応方法**：
1. **影響を評価する**：再開するステージに依存するすべてのステージを特定する
2. **ユーザーに警告する**：「Application Designを再開すると、Units Generation、ユニットごとの設計（すべてのユニット）、Code Generationをやり直す必要があります。よろしいですか？」
3. **明示的な確認を得る**：ユーザーは全体的な影響を理解しなければならない
4. **確認された場合**：
   - 影響を受けるすべての成果物をアーカイブする
   - `aidlc-state.md` で影響を受けるすべてのステージをリセットする
   - 影響を受けるすべての計画ファイルのチェックボックスをクリアする
   - 再開するステージに戻る
   - その時点から前方に向けて再実行する
5. **変更を記録する**：再開の全体的な影響と理由を文書化する

**考慮事項**：
- 大幅な手直しが必要
- すべての依存ステージを再度行わなければならない
- タイムラインが大幅に延長される
- 再開よりも修正の方が適切かどうかを検討する

---

### 5. ステージの深度レベル(depth level)の変更

**シナリオ**：ユーザーが現在または今後のステージの深度レベル(depth level)を変更したい

**例**：「標準ではなく包括的な要件分析を行いましょう」

**対応方法**：
1. **リクエストを確認する**：「Requirements AnalysisをStandardからComprehensiveの深度に変更したいということですね。より徹底的になりますが時間がかかります。よろしいですか？」
2. **実行計画を更新する**：`workflow-planning.md` の深度レベル(depth level)を変更する
3. **アプローチを調整する**：ステージの包括的深度のガイドラインに従う
4. **見積もりを更新する**：新しいタイムライン見積もりをユーザーに伝える
5. **変更を記録する**：深度変更と理由を文書化する

**考慮事項**：
- より深い = より多くの時間だが品質が向上する
- より浅い = より速いが詳細を見落とす可能性がある
- ステージの前または途中でのみ変更できる（完了後は不可）

---

### 6. ワークフローの一時停止

**シナリオ**：ユーザーが一時停止して後で続ける必要がある

**例**：「今は止まって明日続けます」

**対応方法**：
1. **現在のステップを完了する**：可能であれば進行中の現在のステップを終える
2. **チェックボックスを更新する**：完了したすべてのステップを [x] でマークする
3. **状態を更新する**：`aidlc-state.md` が現在の状態を反映していることを確認する
4. **一時停止を記録する**：`audit.md` に一時停止ポイントを文書化する
5. **再開手順を提供する**：「戻ってきたとき、既存のプロジェクトを検出して以下から継続することを提案します：[現在のステージ、現在のステップ]」

**再開時**：
1. **既存プロジェクトを検出する**：`aidlc-state.md` の存在を確認する
2. **コンテキストを読み込む**：完了したステージのすべての成果物を読み込む
3. **状態を表示する**：現在のステージと次のステップを表示する
4. **選択肢を提示する**：中断した箇所から継続するか以前の作業をレビューするか
5. **再開を記録する**：`audit.md` に再開ポイントを文書化する

---

### 7. アーキテクチャの決定の変更

**シナリオ**：ユーザーがモノリスからマイクロサービスへ（またはその逆）変更したい

**例**：「やっぱり、モノリスではなくマイクロサービスにしましょう」

**対応方法**：
1. **現在の進捗を評価する**：ワークフロー(workflow)のどの段階にいるかを判断する
2. **影響を説明する**：
   - Units Generation前の場合：影響は最小限、決定を更新するだけ
   - Units Generation後の場合：Units Generationとすべてのユニットごとの設計を再実行する必要がある
   - Code Generation後の場合：大幅な手直しが必要
3. **アプローチを推奨する**：
   - ワークフローの初期段階：Application Designステージから再開する
   - ワークフローの後期段階：修正が再開と比べて実現可能かどうかを検討する
4. **確認を得る**：ユーザーは変更の全体的な範囲を理解しなければならない
5. **変更を実行する**：影響を受けるステージの再開手順に従う

**考慮事項**：
- アーキテクチャの変更は連鎖的な影響がある
- ワークフローの早い段階ほど変更が容易
- ワークフローの後期ほどコストと効果を考慮する

---

### 8. ユニットの追加/削除

**シナリオ**：ユーザーがUnits Generation後にユニットを追加または削除したい

**例**：「Paymentユニットを PaymentとBillingに分割する必要がある」

**対応方法**：
1. **影響を評価する**：設計やコードが完了しているユニットを特定する
2. **結果を説明する**：
   - ユニットを追加する場合：新しいユニットの完全な設計とコードが必要
   - ユニットを削除する場合：機能を他のユニットに再配分する必要がある
   - ユニットを分割する場合：結果として生じる両方のユニットの設計とコードを再実行する必要がある
3. **ユニット成果物を更新する**：
   - `unit-of-work.md` を修正する
   - `unit-of-work-dependency.md` を更新する
   - `unit-of-work-story-map.md` を修正する
4. **影響を受けるユニットをリセットする**：影響を受けるユニットを再設計が必要としてマークする
5. **変更を実行する**：影響を受けるユニットに対して通常のユニット設計とコードプロセスに従う

**考慮事項**：
- それらのユニットのすべての下流ステージに影響する
- 依存関係が変わると他のユニットにも影響する場合がある
- タイムラインへの影響は影響を受けるユニットの数による

## 変更処理の一般的なガイドライン

### 変更を行う前に

1. **リクエストを理解する**：ユーザーが何を変更したいか、なぜ変更したいかについて確認の質問をする
2. **影響を評価する**：影響を受けるすべてのステージ、成果物、依存関係を特定する
3. **結果を説明する**：何を再実行する必要があるか、タイムラインへの影響を明確に伝える
4. **代替案を提示する**：再開よりも修正の方が適切な場合がある
5. **明示的な確認を得る**：ユーザーは影響を理解して受け入れなければならない

### 変更中

1. **既存の作業をアーカイブする**：破壊的な変更を行う前に必ずバックアップを取る
2. **すべての追跡情報を更新する**：`aidlc-state.md`、計画ファイル、`audit.md` を同期させる
3. **進捗を伝える**：何が起きているかをユーザーに知らせ続ける
4. **変更を検証する**：変更がすべての成果物にわたって一貫していることを確認する
5. **継続性をテストする**：変更後もワークフロー(workflow)がスムーズに続けられることを確認する

### 変更後

1. **一貫性を確認する**：すべての成果物が変更と整合していることを確認する
2. **ドキュメントを更新する**：すべての参照が更新されていることを確認する
3. **完全に記録する**：`audit.md` に変更の完全な履歴を文書化する
4. **ユーザーに確認する**：変更がユーザーの期待を満たしているかを確認する
5. **ワークフローを再開する**：新しい状態から通常の実行を継続する

## 変更リクエストの決定ツリー

```
User requests change
    |
    ├─ Is it current stage?
    |   ├─ Yes: Can modify or restart current stage
    |   └─ No: Go to next question
    |
    ├─ Is it a completed stage?
    |   ├─ Yes: Assess impact on dependent stages
    |   |   ├─ Low impact: Modify and update dependents
    |   |   └─ High impact: Recommend restart from that stage
    |   └─ No: Go to next question
    |
    ├─ Is it adding a skipped stage?
    |   ├─ Yes: Check prerequisites, add to plan, execute
    |   └─ No: Go to next question
    |
    ├─ Is it skipping a planned stage?
    |   ├─ Yes: Warn about impact, get confirmation, skip
    |   └─ No: Go to next question
    |
    └─ Is it changing depth level?
        ├─ Yes: Update plan, adjust approach
        └─ No: Clarify request with user
```

## ベストプラクティス

1. **必ず確認する**：明示的なユーザーの確認なしに破壊的な変更を行わない
2. **影響を説明する**：ユーザーは決定前に結果を理解する必要がある
3. **選択肢を提示する**：変更を処理する方法が複数ある場合がある
4. **先にアーカイブする**：破壊的な変更を行う前に必ずバックアップを取る
5. **すべてを更新する**：すべての追跡ファイルを同期させる
6. **徹底的に記録する**：監査証跡のためにすべての変更を文書化する
7. **後で検証する**：ワークフロー(workflow)がスムーズに継続できることを確認する
8. **柔軟に対応する**：ワークフロー(workflow)はユーザーのニーズに適応するべきであり、硬直したプロセスを強要してはならない
