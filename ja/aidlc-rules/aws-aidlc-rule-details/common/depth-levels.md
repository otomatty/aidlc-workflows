# アダプティブな深度

**目的**：AI-DLC(AI駆動開発ライフサイクル)が問題の複雑さに応じて詳細レベルを適応させる仕組みを説明する

## 基本原則

**ステージが実行されると、そのステージで定義されているすべての成果物が作成されます。「深度」とは、それらの成果物における詳細さと厳密さのレベルを指し、問題の複雑さに応じて適応します。**

## ステージ選択と詳細レベル

### ステージ選択（二値的）
- **ワークフロー計画**が決定する：各ステージについて「実行」または「スキップ」
- **実行の場合**：ステージが動作し、定義されたすべての成果物を作成する
- **スキップの場合**：ステージはまったく実行されない

### 詳細レベル（適応的）
- **シンプルな問題**：必要最低限の詳細を含む簡潔な成果物
- **複雑な問題**：充実した詳細を含む包括的な成果物
- **モデルが判断する**：規定のルールではなく、問題の特性に基づいて決定する

## 詳細レベルに影響する要素

モデルは適切な詳細レベルを決定する際に、以下の要素を考慮します：

1. **リクエストの明確さ**：ユーザーのリクエストはどれだけ明確・完全か？
2. **問題の複雑さ**：ソリューション空間はどれほど複雑か？
3. **スコープ**：単一ファイル、コンポーネント、複数コンポーネント、システム全体のどれか？
4. **リスクレベル**：エラーや漏れによる影響はどの程度か？
5. **利用可能なコンテキスト**：グリーンフィールドかブラウンフィールドか、既存ドキュメントの有無
6. **ユーザーの好み**：簡潔さや詳細さについてユーザーが希望を示しているか？

## 例：要件分析の成果物

**すべてのシナリオで同じ成果物を作成する**：
- `requirement-verification-questions.md`（必要な場合）
- `requirements.md`

**注意**：ユーザーの最初のリクエストは `audit.md` に記録されます（別途 user-intent.md は不要）

**詳細レベルは複雑さによって異なる**：

### シンプルなシナリオ（バグ修正）
- **requirement-verification-questions.md**：必要な確認の質問
- **requirements.md**：簡潔な機能要件、最小限のセクション

### 複雑なシナリオ（システム移行）
- **requirement-verification-questions.md**：複数ラウンド、10件以上の質問
- **requirements.md**：包括的な機能要件と非機能要件、トレーサビリティ、受け入れ基準

## 例：アプリケーション設計の成果物

**すべてのシナリオで同じ成果物を作成する**：
- `application-design.md`
- `component-diagram.md`

**詳細レベルは複雑さによって異なる**：

### シンプルなシナリオ（単一コンポーネント）
- **application-design.md**：基本的なコンポーネントの説明、主要なメソッド
- **component-diagram.md**：必要な関係性のみを示したシンプルな図

### 複雑なシナリオ（マルチコンポーネントシステム）
- **application-design.md**：詳細なコンポーネントの責務、すべてのメソッドとシグネチャ、デザインパターン、検討した代替案
- **component-diagram.md**：すべての関係性、データフロー、統合ポイントを示した包括的な図

## モデルへの指針

**「その問題に必要な詳細をちょうど作成する―多くでも少なくでもなく。」**

- シンプルな問題に不必要な詳細を加えて人工的に膨らませない
- 複雑な問題から重要な詳細を省略して不十分な内容にしない
- 問題の特性が自然に詳細レベルを導くようにする
- ステージが実行されるとき、必要なすべての成果物を必ず作成する
