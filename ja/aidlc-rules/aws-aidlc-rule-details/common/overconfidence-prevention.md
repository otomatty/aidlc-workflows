# 過信防止ガイド

## 問題の概要

AI-DLC(AI駆動開発ライフサイクル)は、複雑なプロジェクトの意図の記述に対しても確認の質問を十分に行わないという過信の傾向を示していた。その結果、適切な要件収集を行う代わりに仮定に基づいて進めてしまうケースが発生していた。

## 根本原因の分析

過信の問題は、複数のステージにおいて質問のスキップを促すディレクティブが存在することが原因だった：

1. **Functional Design**：「該当しない場合はカテゴリ全体をスキップする」
2. **User Stories**：「カテゴリはインスピレーションとして使用し、必須のチェックリストとは見なさない」
3. **Requirements Analysis**：質問を最小化することを促す同様のパターン
4. **NFR Requirements**：徹底した分析を妨げる「only if（条件付き）」条件

これらのディレクティブは、包括的な要件収集を促進するのではなく、AIに質問を避けるよう指示していた。

## 実施した解決策

### 質問生成の哲学の更新

**旧アプローチ**：「絶対に必要な場合にのみ質問する」
**新アプローチ**：「疑わしいときは質問する―過信は悪い結果につながる」

### 主な変更点

#### 1. 要件分析ステージ
- 「必要な場合のみ」から「例外的に明確でない限り必ず質問を作成する」に変更
- 包括的な評価領域を追加（機能的、非機能的、ビジネスコンテキスト、技術的コンテキスト）
- 積極的な質問アプローチを強調

#### 2. ユーザーストーリーステージ
- 「カテゴリ全体をスキップする」ディレクティブを削除
- 評価すべき包括的な質問カテゴリを追加
- 回答分析の要件を強化
- フォローアップ質問の義務を強化

#### 3. Functional Designステージ
- 「only if」条件を包括的な評価に置き換え
- 質問カテゴリを追加（データフロー、統合ポイント、エラー処理）
- 曖昧さの検出と解消要件を強化

#### 4. NFR Requirementsステージ
- 基本的なNFRを超えた質問カテゴリを拡大
- 信頼性、保守性、ユーザビリティの考慮事項を追加
- 技術的な曖昧さに対する回答分析を強化

### 新しい指針原則

1. **デフォルトで質問する**：曖昧さがあれば確認の質問をする
2. **包括的なカバレッジ**：関連するすべてのカテゴリを評価し、領域をスキップしない
3. **徹底的な分析**：すべてのユーザー回答を注意深く分析し曖昧さを見つける
4. **フォローアップの義務化**：不明確な回答に対してはフォローアップ質問を作成する
5. **曖昧さがある状態での進行禁止**：すべての曖昧さが解消されるまで先に進まない

## 実装ガイドライン

### 質問生成について
- すべての質問カテゴリを評価し、どれもスキップしない
- 明確にすることで品質が向上する箇所であれば質問する
- 各ステージに包括的な質問カテゴリを含める
- 質問の除外ではなく含めることをデフォルトとする

### 回答分析について
- 曖昧な回答を見つける：「場合による」「かもしれない」「わからない」「混在している」「どちらかといえば」
- 未定義の用語や外部概念への参照を検出する
- 矛盾した、または不完全な回答を特定する
- 曖昧さがあればフォローアップ質問を作成する

### フォローアップ質問について
- 曖昧さが検出された場合は別途確認ファイルを作成する
- 各曖昧さを解消するための具体的な質問をする
- すべての不明確な回答が明確化されるまで進めない
- 徹底的に行う―過剰な確認は不十分な確認よりも望ましい

## 品質保証

### 注意すべきレッドフラグ
- 複雑なプロジェクトでどの質問もしないままステージが完了する
- 曖昧またはあいまいなユーザー回答のまま進行する
- 正当な理由なく質問カテゴリ全体をスキップする
- 明確化を求める代わりに仮定を行う

### 成功の指標
- プロジェクトの複雑さに見合った適切な数の確認質問
- 必要に応じてフォローアップを含むユーザー回答の徹底的な分析
- 実装に進む前の明確で曖昧さのない要件
- 事前の明確化が十分であることによる後のステージでの変更の減少

## メンテナンス

このガイドは以下の場合に参照すること：
- AI-DLC(AI駆動開発ライフサイクル)に新しいステージを追加する場合
- 既存のステージの指示を更新する場合
- 過信の問題に対するAI-DLCのパフォーマンスをレビューする場合
- 質問生成の原則についてチームメンバーをトレーニングする場合

## 重要なポイント

**誤った仮定を行うよりも、多くの質問をする方が望ましい。** 事前に確認の質問をするコストは、仮定に基づいて間違ったソリューションを実装するコストよりもはるかに小さい。
