# エラー処理と復旧手順

## 一般的なエラー処理の原則

### エラーが発生した場合
1. **エラーを特定する**：何が問題だったかを明確に述べる
2. **影響を評価する**：エラーがブロッキングかどうか、または回避策があるかを判断する
3. **コミュニケーションを取る**：ユーザーにエラーと選択肢について知らせる
4. **解決策を提示する**：エラーを解消または回避するための明確な手順を提供する
5. **記録する**：エラーと解決策を `audit.md` に記録する

### エラーの重大度レベル

**Critical（致命的）**：ワークフローを継続できない
- 必須ファイルまたは成果物が存在しない
- 処理できない無効なユーザー入力
- ファイル操作を妨げるシステムエラー

**High（高）**：ステージが予定どおりに完了できない
- 必須質問への回答が不完全
- ユーザーの回答が矛盾している
- 前のステージからの依存関係が欠如している

**Medium（中）**：回避策によりステージを継続できる
- オプションの成果物が欠如している
- 非致命的な検証の失敗
- 部分的な完了が可能

**Low（低）**：進行を妨げない軽微な問題
- フォーマットの不一致
- オプション情報の欠如
- ブロックしない警告

## ステージ別のエラー処理

### ワークスペース検出エラー

**エラー**：ワークスペースファイルを読み取れない
- **原因**：権限の問題、ディレクトリが存在しない
- **解決策**：ワークスペースのパスと権限を確認するようユーザーに依頼する
- **回避策**：ユーザーが提供した情報のみで進める

**エラー**：既存の `aidlc-state.md` が破損している
- **原因**：手動編集、前回の実行が不完全
- **解決策**：最初からやり直すか復旧を試みるかをユーザーに確認する
- **復旧**：バックアップを作成し、新しい状態ファイルを開始する

**エラー**：必要なステージを決定できない
- **原因**：ユーザーからの情報が不十分
- **解決策**：意図とスコープについて確認の質問をする
- **回避策**：包括的な実行計画をデフォルトとして使用する

### 要件分析エラー

**エラー**：ユーザーが矛盾する要件を提示する
- **原因**：理解が不明確、ニーズが変化している
- **解決策**：矛盾を解消するためのフォローアップ質問を作成する
- **進行不可**：矛盾が解消されるまで進めない

**エラー**：要件ドキュメントを変換できない
- **原因**：サポートされていないフォーマット、ファイルが破損している
- **解決策**：サポートされているフォーマットで要件を提供するようユーザーに依頼する
- **回避策**：ユーザーの口頭説明をもとに作業する

**エラー**：検証質問への回答が不完全
- **原因**：ユーザーが質問をスキップした、何を回答すべきか不明
- **解決策**：未回答の質問を強調し、例を提供する
- **進行不可**：必須の質問すべてに回答が得られるまで進めない

### ユーザーストーリーエラー

**エラー**：要件をストーリーに対応付けられない
- **原因**：要件が曖昧すぎる、機能的な詳細が欠如している
- **解決策**：明確化のために要件分析に戻る
- **回避策**：利用可能な情報に基づいてストーリーを作成し、不完全とマークする

**エラー**：ユーザーがストーリー計画の回答で曖昧な内容を提示する
- **原因**：選択肢が不明確、複雑な判断が必要
- **解決策**：具体的な例を含むフォローアップ質問を追加する
- **進行不可**：曖昧さが解消されるまで進めない

**エラー**：ストーリー生成計画に未完了のステップがある
- **原因**：実行が中断された、ステップがスキップされた
- **解決策**：最初の未完了ステップから再開する
- **復旧**：完了したステップを確認し、チェックポイントから継続する

### アプリケーション設計エラー

**エラー**：アーキテクチャの決定が不明確または矛盾している
- **原因**：曖昧な回答、要件が競合している
- **解決策**：決定を明確にするためのフォローアップ質問を追加する
- **進行不可**：決定が明確化・文書化されるまで進めない

**エラー**：サービス/ユニットの数を決定できない
- **原因**：境界についての情報が不十分
- **解決策**：デプロイメント、チーム構造、スケーリングに関する具体的な質問をする
- **回避策**：モノリスをデフォルトとし、後で変更できるようにする

### 設計エラー

**エラー**：ユニットの依存関係が循環している
- **原因**：境界定義が不適切、密結合
- **解決策**：循環依存を特定し、リファクタリングを提案する
- **復旧**：循環を断ち切るようにユニット境界を修正する

**エラー**：ユニット設計計画にステップが欠落している
- **原因**：計画の生成が不完全、テンプレートエラー
- **解決策**：必要なすべてのステップを含む計画を再生成する
- **復旧**：既存の計画に欠落しているステップを追加する

**エラー**：設計成果物を生成できない
- **原因**：ユニット情報が欠如している、要件が不明確
- **解決策**：ユニット定義を明確化するためにユニット生成に戻る
- **回避策**：部分的な設計を生成し、ギャップをマークする

### NFR実装エラー

**エラー**：技術スタックの選択に互換性がない
- **原因**：要件が競合している、プラットフォームの制限
- **解決策**：非互換性を強調し、ユーザーに選択を求める
- **進行不可**：互換性のある選択が行われるまで進めない

**エラー**：組織上の制約を満たせない
- **原因**：ネットワーク制限、セキュリティポリシー
- **解決策**：制約を文書化し、回避策をユーザーに確認する
- **エスカレーション**：セットアップに人間の介入が必要な場合がある

**エラー**：NFR実装ステップで人間のアクションが必要
- **原因**：AIが実行できないタスクがある（ネットワーク設定、認証情報など）
- **解決策**：**HUMAN TASK** として明確にマークし、手順を提供する
- **待機**：続行前にユーザーの確認を待つ

### コード生成計画エラー

**エラー**：コード生成計画が不完全
- **原因**：設計成果物が欠如している、要件が不明確
- **解決策**：設計ステージに戻って成果物を完成させる
- **復旧**：利用可能な情報で計画を生成し、ギャップをマークする

**エラー**：ユニットの依存関係が満たされていない
- **原因**：依存するユニットがまだ生成されていない
- **解決策**：依存関係を尊重するように生成シーケンスを並べ替える
- **回避策**：スタブ依存関係で生成し、後で統合する

### コード生成エラー（パート2：コード生成）

**エラー**：あるステップのコードを生成できない
- **原因**：設計情報が不十分、要件が不明確
- **解決策**：ステップをスキップし、不完全と記録して継続する
- **復旧**：情報を追加収集した後、そのステップに戻る

**エラー**：生成されたコードに構文エラーがある
- **原因**：テンプレートの問題、言語固有の問題
- **解決策**：構文エラーを修正し、必要であれば再生成する
- **検証**：続行前にコードがコンパイルできることを確認する

**エラー**：テスト生成が失敗する
- **原因**：複雑なロジック、テストフレームワークの設定が欠如している
- **解決策**：基本的なテスト構造を生成し、手動完了としてマークする
- **回避策**：テストなしで進み、Operationsフェーズで追加する

### Operationsエラー

**エラー**：ビルドツールを特定できない
- **原因**：通常とは異なるプロジェクト構造、複数のビルドシステム
- **解決策**：ビルドツールとコマンドを指定するようユーザーに依頼する
- **回避策**：汎用的な手順を提供し、ユーザーが適応する

**エラー**：デプロイメント対象が不明確
- **原因**：複数の環境、複雑なインフラ
- **解決策**：デプロイメント対象と方法を指定するようユーザーに依頼する
- **回避策**：一般的なプラットフォーム向けの手順を提供する

## 復旧手順

### ステージの部分的な完了

**シナリオ**：ステージが実行途中で中断された

**復旧手順**:
1. ステージ計画ファイルを読み込む
2. 最後に完了したステップを特定する（最後の [x] チェックボックス）
3. 次の未完了ステップから再開する
4. 前のステップがすべて実際に完了しているか確認する
5. 通常どおり実行を継続する

### 状態ファイルの破損

**シナリオ**：`aidlc-state.md` が破損または不整合

**復旧手順**:
1. バックアップを作成する：`aidlc-state.md.backup`
2. どのステージにいるかをユーザーに確認する
3. 状態ファイルをゼロから再生成する
4. 既存の成果物に基づいて完了したステージをマークする
5. 現在のステージから再開する

### 成果物の欠如

**シナリオ**：前のステージからの必須成果物が欠如している

**復旧手順**:
1. 欠落している成果物を特定する
2. 再生成できるかどうかを判断する
3. 可能な場合：そのステージに戻り、成果物を再生成する
4. 不可能な場合：情報を手動で提供するようユーザーに依頼する
5. `audit.md` にギャップを文書化する

### ユーザーがステージの再開を希望する場合

**シナリオ**：ユーザーがステージの結果に不満でやり直したい

**復旧手順**:
1. 再開を確認する（データが失われる旨を伝える）
2. 既存の成果物をアーカイブする：`{artifact}.backup`
3. `aidlc-state.md` のステージ状態をリセットする
4. 計画ファイルのステージチェックボックスをクリアする
5. ステージを最初から再実行する

### ユーザーがステージのスキップを希望する場合

**シナリオ**：ユーザーが計画されていたステージをスキップしたい

**復旧手順**:
1. 影響をユーザーが理解しているか確認する
2. スキップの理由を `audit.md` に文書化する
3. `aidlc-state.md` でステージを「SKIPPED」としてマークする
4. 次のステージに進む
5. 注意：依存関係が欠如している場合、後のステージで問題が発生する可能性がある

## エスカレーションのガイドライン

### ユーザーの助けを求めるタイミング

**即座に**:
- ユーザーの入力が矛盾または曖昧
- 必須情報が欠落している
- AIが解決できない技術的制約
- ビジネス判断が必要な決定

**解決を試みた後**:
- 同じステップで繰り返しエラーが発生する
- 複雑な技術的問題
- 通常とは異なるプロジェクト構造
- 外部システムとの統合

### 最初からやり直しを提案するタイミング

**以下の場合はやり直しを検討する**:
- 複数のステージでエラーが発生している
- 状態ファイルが深刻に破損している
- ユーザーの要件が大幅に変更された
- アーキテクチャの決定を覆す必要がある
- ユーザーが欠落した情報を提供できない
- 成果物がフェーズ間で一貫していない

**やり直す前に**:
1. 既存の作業をすべてアーカイブする
2. 学んだ教訓を文書化する
3. 保存するものを特定する
4. ユーザーの確認を得る
5. 新しい実行計画を作成する

## セッション再開時のエラー

### 再開時の成果物の欠如

**エラー**：前のステージからの必須成果物が欠落している
- **原因**：ファイルが削除、移動、または作成されなかった
- **解決策**：
  1. 欠落している成果物を作成したステージを特定する
  2. そのステージが aidlc-state.md で完了としてマークされているか確認する
  3. 完了とマークされているが成果物が欠落している場合：そのステージを再生成する
  4. 完了とマークされていない場合：そのステージから再開する
- **復旧**：欠落している成果物を作成するステージに戻り、再実行する

**エラー**：成果物ファイルは存在するが空または破損している
- **原因**：書き込みの中断、手動編集、ファイルシステムの問題
- **解決策**：
  1. 破損したファイルのバックアップを作成する
  2. それを作成したステージから再生成を試みる
  3. 再生成できない場合：情報を再作成するためにユーザーに確認する
- **復旧**：その成果物を作成するステージを再実行する

### 再開時の状態の不整合

**エラー**：aidlc-state.md がステージを完了とマークしているが成果物が存在しない
- **原因**：状態ファイルは更新されたが成果物の生成が失敗した
- **解決策**：
  1. aidlc-state.md でそのステージを未完了としてマークする
  2. ステージを再実行して成果物を生成する
  3. 完了とマークする前に成果物の存在を確認する
- **復旧**：ステージ状態をリセットして再実行する

**エラー**：成果物は存在するが aidlc-state.md がステージを未完了としてマークしている
- **原因**：成果物の生成は成功したが状態の更新が失敗した
- **解決策**：
  1. 成果物が完全で有効であることを確認する
  2. aidlc-state.md を更新してステージを完了としてマークする
  3. 次のステージに進む
- **復旧**：実際の完了状態を反映するよう状態ファイルを更新する

**エラー**：aidlc-state.md で複数のステージが「current（現在）」としてマークされている
- **原因**：状態ファイルの破損、手動編集
- **解決策**：
  1. 実際の進捗を判断するために成果物を確認する
  2. どのステージにいるかをユーザーに確認する
  3. aidlc-state.md を修正して単一の現在ステージを示す
- **復旧**：既存の成果物に基づいて状態ファイルを再構築する

### コンテキスト読み込みエラー

**エラー**：前のステージから必要なコンテキストを読み込めない
- **原因**：ファイルが欠落、コンテンツが破損、ファイルパスが誤っている
- **解決策**：
  1. 現在のステージに必要な成果物を一覧表示する
  2. 欠落または破損しているものを確認する
  3. 欠落している成果物を再生成するか、情報をユーザーに確認する
- **復旧**：現在のステージを再開する前に前提条件のステージを完成させる

**エラー**：読み込んだ成果物に矛盾する情報が含まれている
- **原因**：手動編集、複数人での作業、不完全な更新
- **解決策**：
  1. 矛盾を特定してユーザーに提示する
  2. どの情報が正しいかをユーザーに確認する
  3. 矛盾を解消するよう成果物を更新する
- **復旧**：続行する前に矛盾を調整する

### 再開のベストプラクティス

1. **状態を必ず検証する**：aidlc-state.md が実際の成果物と一致することを確認する
2. **段階的に読み込む**：ステージごとに成果物を読み込み、それぞれを検証する
3. **早めに失敗する**：重要な成果物が欠落している場合はすぐに停止する
4. **明確に伝える**：何が欠落していて、なぜ必要かをユーザーに正確に伝える
5. **選択肢を提示する**：再生成、手動提供、または最初からやり直し
6. **復旧を記録する**：すべての復旧アクションを audit.md に記録する

## ログ記録の要件

### エラーログのフォーマット

```markdown
## Error - [Stage Name]
**Timestamp**: [ISO timestamp]
**Error Type**: [Critical/High/Medium/Low]
**Description**: [What went wrong]
**Cause**: [Why it happened]
**Resolution**: [How it was resolved]
**Impact**: [Effect on workflow]

---
```

### 復旧ログのフォーマット

```markdown
## Recovery - [Stage Name]
**Timestamp**: [ISO timestamp]
**Issue**: [What needed recovery]
**Recovery Steps**: [What was done]
**Outcome**: [Result of recovery]
**Artifacts Affected**: [List of files]

---
```

## 予防のベストプラクティス

1. **早めに検証する**：作業を開始する前に入力と依存関係を確認する
2. **こまめにチェックポイントを設ける**：ステップを完了した直後にチェックボックスを更新する
3. **明確に伝える**：何を行っているか、なぜ行っているかを説明する
4. **質問する**：推測しない―曖昧さはすぐに明確化する
5. **すべてを文書化する**：すべての決定と変更を `audit.md` に記録する
