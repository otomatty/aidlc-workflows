# 技術環境ドキュメント：CalcEngine 関数電卓 API

## プロジェクト技術サマリー

- **プロジェクト名**: CalcEngine
- **プロジェクト種別**: グリーンフィールド
- **主要ランタイム環境**: クラウド
- **クラウドプロバイダー**: AWS
- **デプロイメントモデル**: サーバーレス（API Gateway + Lambda）
- **パッケージマネージャー**: uv
- **チーム規模**: 4名（バックエンド開発者2名、ドキュメントポータル担当フロントエンド開発者1名、QAエンジニア1名）
- **チームの経験**: Pythonバックエンドの豊富な経験、AWSの中程度の経験、数学ライブラリ開発の経験なし。FastAPIとFlaskを業務で使用した経験あり。pytestに習熟。CDKの経験は限定的（例が必要）。

## パッケージおよび環境管理

### 標準ツールとしてのuv

uvはこのプロジェクトの**唯一のパッケージおよび環境管理ツール**です。pip、pip-tools、poetry、pipenv、condaは使用しないこと。

### uvの使用標準

```bash
# プロジェクトの初期化（すでに完了済み。再実行しないこと）
uv init calcengine
cd calcengine

# 依存関係の追加
uv add fastapi                      # ランタイム依存関係の追加
uv add uvicorn[standard]            # エクストラ付きで追加
uv add --dev pytest pytest-cov      # 開発用依存関係の追加
uv add --dev mypy ruff              # 開発ツールの追加

# 依存関係の削除
uv remove requests                  # 依存関係の削除

# プロジェクト環境でのコマンド実行
uv run python -m calcengine.main    # アプリケーションの実行
uv run pytest                       # テストの実行
uv run mypy src/                    # 型チェックの実行
uv run ruff check src/              # リンターの実行

# ロックファイルから環境を同期
uv sync                             # uv.lockからすべての依存関係をインストール
uv sync --dev                       # 開発用依存関係も含める

# ロックファイルの管理
# uv.lockは自動生成される。手動で編集しないこと。
# uv.lockはバージョン管理にコミットしなければならない。
```

### 依存ファイルの標準

| ファイル              | 用途                                                          | Gitにコミット  |
| ----------------- | ------------------------------------------------------------- | ----------------- |
| `pyproject.toml`  | プロジェクトメタデータ、依存関係の宣言、ツール設定             | する               |
| `uv.lock`         | 正確な解決済みバージョンを含む決定論的ロックファイル           | する               |

## 推奨技術とパターン

### アーキテクチャパターン

**サーバーレス関数としてデプロイするモジュラーモノリス。**

CalcEngineは内部モジュール（算術、三角法、統計など）を持つ単一のPythonパッケージで、単一のFastAPIアプリケーションを通じて公開され、API Gatewayの背後のAWS Lambdaにデプロイされます。これはマイクロサービスアーキテクチャではありません。

| 決定事項           | 選択                                                     | 理由                                                                                                                                                                        |
| ------------------ | -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| アーキテクチャスタイル | モジュラーモノリス                                     | 小規模チーム（4名）、単一ドメイン、MVPではモジュールごとの独立したスケーリング要件なし。                                                                                 |
| デプロイメントモデル | Mangum経由でAPIルートをすべて提供する単一Lambda関数      | シンプルさ。単一のデプロイアーティファクト。コールドスタートがすべてのエンドポイントにまたがって均等化される。                                                            |
| モジュール境界      | `src/calcengine/` 内のPythonパッケージ                  | 個別サービスの運用コストなしに明確な内部境界。特定のエンドポイントが異なるメモリ/タイムアウトを必要とする場合は後から個別のLambdaに切り出せる。                          |

### API設計標準

- **スタイル**: HTTPS上のREST。JSONリクエストおよびレスポンスボディ。
- **ベースURL**: `https://api.calcengine.io/v1/`
- **バージョニング**: URLパスプレフィックス（`/v1/`、`/v2/`）。メジャーバージョンのみ。後方互換性のある変更はバージョンを上げない。
- **ドキュメント**: FastAPIが自動生成するOpenAPI 3.1仕様。`https://docs.calcengine.io` でホスト。
- **命名規則**: JSONフィールド名はsnake_case（Pythonの慣習）。URLパスはkebab-case。
- **コンテンツタイプ**: すべてのリクエストとレスポンスに `application/json`。XMLサポートなし。

**標準リクエストフォーマット：**

```json
{
  "expression": "sin(pi/4) * 2 + sqrt(16)",
  "options": {
    "angle_mode": "radians",
    "precision": 15
  }
}
```

**標準成功レスポンスフォーマット：**

```json
{
  "result": 5.414213562373095,
  "expression": "sin(pi/4) * 2 + sqrt(16)",
  "computation_time_ms": 2.3,
  "engine_version": "0.1.0"
}
```

**標準エラーレスポンスフォーマット：**

```json
{
  "error": {
    "code": "DOMAIN_ERROR",
    "message": "Cannot compute logarithm of a negative number",
    "detail": "log(-5) is undefined for real numbers",
    "parameter": "expression",
    "documentation_url": "https://docs.calcengine.io/errors/DOMAIN_ERROR"
  }
}
```

**エラーコード（MVP）：**

| コード                  | HTTPステータス | 意味                                                            |
| --------------------- | ------------ | --------------------------------------------------------------- |
| `PARSE_ERROR`         | 400          | 式を解析できなかった。不正な構文。                                |
| `DOMAIN_ERROR`        | 422          | 数学的に未定義（log(-1)、sqrt(-1)、ゼロ除算）。                  |
| `OVERFLOW_ERROR`      | 422          | 結果が表現可能な範囲を超えた。                                   |
| `INVALID_PARAMETER`   | 400          | リクエストパラメーターの型または値が無効。                        |
| `EXPRESSION_TOO_LONG` | 400          | 式が許可されている最大長を超えた。                               |
| `RATE_LIMIT_EXCEEDED` | 429          | APIキーがレート制限を超えた。                                    |
| `UNAUTHORIZED`        | 401          | APIキーがないか無効。                                            |
| `INTERNAL_ERROR`      | 500          | 予期しないサーバーエラー。                                       |

### データパターン

- **主要データストア**: DynamoDB（シングルテーブル設計）
- **DynamoDBのエンティティ**: APIキー、使用量カウンター（キーごと・月ごと）、レート制限ウィンドウ（キーごと・分ごと）
- **アクセスパターン**: すべての読み取り・書き込みはプライマリキー（APIキーID）で行う。スキャンなし。複雑なクエリなし。
- **キャッシュ**: 外部キャッシュなし。LambdaはウォームなInvocation間でDynamoDB接続を再利用する。APIキーの検証結果はLambdaメモリに60秒キャッシュされる。
- **リレーショナルデータベースなし**: リレーショナルクエリが必要になった場合（レポート、分析）、RDSを追加する前にDynamoDBのエクスポート to S3 + Athenaを評価すること。

### ログパターン

すべてのログ出力はstructlog経由で構造化JSONでなければならない。人間が読みやすいコンソール出力はローカル開発のみ。

```python
import structlog

logger = structlog.get_logger()

# 標準ログ呼び出し
logger.info(
    "calculation_completed",
    expression=expression,
    result=result,
    computation_time_ms=elapsed,
    api_key_id=api_key_id,
)

# エラーログ呼び出し
logger.error(
    "calculation_failed",
    expression=expression,
    error_code="DOMAIN_ERROR",
    error_detail=str(e),
    api_key_id=api_key_id,
)
```

## セキュリティ要件

### 認証と認可

- **APIコール認証**: `Authorization: Bearer {key}` ヘッダーで渡されるAPIキー。APIキーは32文字のランダムな文字列で、DynamoDBにbcryptハッシュとして保存される。
- **開発者ポータル認証**: Amazon Cognitoユーザープール。メール確認を伴うメール+パスワードのサインアップ。
- **認可モデル**: フラット。すべてのAPIキーはすべてのエンドポイントにアクセスできる。ティアベースのレート制限（無料、スターター、プロフェッショナル）はエンドポイントレベルの権限ではなく使用量計測で強制される。
- **APIキー管理**: 開発者は開発者ポータルでキーの作成、ローテーション、失効を行う。アカウントごとに最大3つのアクティブなキー。

### データ保護

- **保存時の暗号化**: DynamoDBはAWS管理のKMSキーで暗号化。S3バケットはSSE-S3で暗号化。
- **転送時の暗号化**: API Gatewayのカスタムドメインとの CloudFrontディストリビューションにTLS 1.2以上を強制。HTTP（平文）エンドポイントなし。
- **PIIの取り扱い**: 開発者アカウントはメールとハッシュ化されたパスワードを保存。他のPIIは収集しない。数学的な式はPIIではない。式はデバッグのためにログに記録されるが、永続的には保存されない（CloudWatchのログ保持期間：30日）。
- **データ分類**: APIキー = 機密。開発者のメール = 内部。数学的な式と結果 = パブリック。

### 入力バリデーション

- **式の長さ制限**: 最大4,096文字。長い式は `EXPRESSION_TOO_LONG` で拒否する。
- **式の文字ホワイトリスト**: 英数字、算術演算子（`+ - * / ^ %`）、括弧、小数点、カンマ、空白、および認識された関数名。認識されない文字は拒否する。
- **コード実行なし**: 式パーサーは `eval()`、`exec()`、`compile()`、またはいかなる動的コード実行も呼び出してはならない。式はASTに解析され、数学エンジンによって評価される。
- **再帰深度制限**: 式パーサーはネストの深さを100レベルに制限する。`(((((...))))` のような深くネストされた式でのスタックオーバーフローを防ぐ。
- **数値範囲バリデーション**: IEEE 754倍精度範囲を超える結果は `Infinity` や `NaN` の代わりに `OVERFLOW_ERROR` を返す。

### シークレット管理

- **Stripe APIキー**: AWS Secrets Managerに保存。Lambdaはコールドスタート時に読み込み、メモリにキャッシュ。
- **Cognitoクライアントシークレット**: AWS Secrets Managerに保存。
- **禁止されているプラクティス**:
  - `pyproject.toml`、ソースコード、またはGitにコミットされた `.env` ファイルにシークレットを含めない
  - Lambda環境変数にシークレットを含めない（ランタイムでSecretsManagerを使用）
  - コードにAWSアクセスキーを含めない（LambdaはIAM実行ロールを使用）
  - `.env` ファイルはローカル開発のみに使用し、`.gitignore` に記載する

### 依存関係のセキュリティ

- **スキャン**: GitHub DependabotをPythonの依存関係に対して有効化。既知の脆弱性についてのアラートを発する。
- **ライセンスポリシー**: 許可：MIT、Apache 2.0、BSD（2条項および3条項）、PSF、ISC。禁止：GPL、LGPL、AGPL、SSPL、プロプライエタリ。新しい依存関係を追加する前に `uv tree` で確認する。
- **更新ポリシー**: 重大/高CVEは7日以内にパッチ適用。中は30日以内。低は四半期ごとに評価。

### ユニットテストの標準

- **カバレッジ最小値**: 行カバレッジ90%、ブランチカバレッジ80%。`pyproject.toml` の `fail_under = 90` で `pytest-cov` が強制する。
- **モッキングポリシー**: AWSサービス（DynamoDB、Secrets Manager）はmotoでモック。時間はfreezegunでモック。内部の数学関数はモックしない。数学関数は実際の計算でテストしなければならない。
- **命名規則**: テストファイルはソースファイルを反映する。`src/calcengine/trig.py` は `tests/unit/test_trig.py` でテスト。テスト関数は `test_<function>_<scenario>` と命名（例：`test_sin_zero_returns_zero`、`test_sin_negative_pi_returns_zero`）。
- **テストの場所**: 独立した `tests/` ディレクトリツリー。ソースと同じ場所には置かない。

### 数学的精度テスト

これはCalcEngine固有のテストカテゴリーであり、ほとんどのプロジェクトには存在しません。

- **参照実装**: すべての数学関数はPythonの `math` モジュール、`mpmath` ライブラリ（高精度）、または発表された数学的な表に対してテストしなければならない。
- **hypothesisを使った特性ベーステスト**: hypothesisを使用してランダムな有効入力を生成し、特性が保持されることを確認する（例：`sin(x)^2 + cos(x)^2 == 1`、`log(a*b) == log(a) + log(b)`）。
- **エッジケース**: すべての関数は以下の明示的なテストを持たなければならない：ゼロ、負のゼロ、非常に小さな数（epsilon付近）、非常に大きな数、定義域の境界（例：asin(1)、asin(1.0000001)）、特殊な値（pi、e、三角法のためのpi/2の倍数）。
- **許容誤差**: 結果は基本関数についてはULP（最後の桁の単位）1以内で参照値と一致しなければならない。より広い許容誤差が受け入れられる関数については、正当性と共に文書化すること。

**精度テストのパターン例：**

```python
import math
import pytest
from hypothesis import given, strategies as st
from calcengine.trig import sin, cos

class TestSinAccuracy:
    """Validate sin() accuracy against math.sin and known exact values."""

    @pytest.mark.accuracy
    @pytest.mark.parametrize("input_val, expected", [
        (0, 0.0),
        (math.pi / 6, 0.5),
        (math.pi / 4, math.sqrt(2) / 2),
        (math.pi / 2, 1.0),
        (math.pi, 0.0),
        (3 * math.pi / 2, -1.0),
        (2 * math.pi, 0.0),
        (-math.pi / 2, -1.0),
    ])
    def test_sin_known_values(self, input_val: float, expected: float) -> None:
        result = sin(input_val)
        assert result == pytest.approx(expected, abs=1e-15)

    @pytest.mark.accuracy
    @given(st.floats(min_value=-1e6, max_value=1e6, allow_nan=False, allow_infinity=False))
    def test_sin_matches_stdlib(self, x: float) -> None:
        assert sin(x) == pytest.approx(math.sin(x), rel=1e-15)

    @pytest.mark.accuracy
    @given(st.floats(min_value=-1e6, max_value=1e6, allow_nan=False, allow_infinity=False))
    def test_pythagorean_identity(self, x: float) -> None:
        assert sin(x) ** 2 + cos(x) ** 2 == pytest.approx(1.0, abs=1e-14)
```

```python
"""Example: Standard API endpoint pattern for CalcEngine."""

import time

import structlog
from fastapi import APIRouter, Depends
from pydantic import BaseModel, Field

from calcengine.api.middleware.auth import get_api_key_id
from calcengine.api.models.errors import error_response
from calcengine.api.models.responses import CalculationResponse
from calcengine.engine.errors import CalcEngineError
from calcengine.engine.trigonometry import sin

logger = structlog.get_logger()

router = APIRouter()


class SinRequest(BaseModel):
    """Request model for sine calculation."""

    value: float = Field(..., description="Input angle")
    angle_mode: str = Field(
        default="radians",
        pattern="^(radians|degrees)$",
        description="Angle unit: 'radians' or 'degrees'",
    )


@router.post("/v1/trigonometry/sin", response_model=CalculationResponse)
async def calculate_sin(
    request: SinRequest,
    api_key_id: str = Depends(get_api_key_id),
) -> CalculationResponse | dict:
    """Calculate the sine of the given value."""
    start = time.perf_counter()

    try:
        result = sin(request.value, angle_mode=request.angle_mode)
        elapsed = (time.perf_counter() - start) * 1000

        logger.info(
            "calculation_completed",
            endpoint="/v1/trigonometry/sin",
            input_value=request.value,
            angle_mode=request.angle_mode,
            result=result,
            computation_time_ms=round(elapsed, 3),
            api_key_id=api_key_id,
        )

        return CalculationResponse(
            result=result,
            expression=f"sin({request.value})",
            computation_time_ms=round(elapsed, 3),
        )

    except CalcEngineError as e:
        elapsed = (time.perf_counter() - start) * 1000
        logger.warning(
            "calculation_failed",
            endpoint="/v1/trigonometry/sin",
            input_value=request.value,
            error_code=e.code,
            error_detail=str(e),
            computation_time_ms=round(elapsed, 3),
            api_key_id=api_key_id,
        )
        return error_response(e)
```

`examples/math-function/example_function.py`:

```python
"""Example: Standard pattern for a pure math function in CalcEngine engine layer."""

import math

from calcengine.engine.errors import DomainError


def log_base(value: float, base: float = 10.0) -> float:
    """Compute the logarithm of a value with the given base.

    Args:
        value: The number to compute the logarithm of. Must be positive.
        base: The logarithm base. Must be positive and not equal to 1.
              Defaults to 10 (common logarithm).

    Returns:
        The logarithm of value in the given base.

    Raises:
        DomainError: If value <= 0, base <= 0, or base == 1.
    """
    if value <= 0:
        raise DomainError(
            code="DOMAIN_ERROR",
            message=f"Cannot compute logarithm of {value}",
            detail="Logarithm is only defined for positive numbers",
            parameter="value",
        )

    if base <= 0:
        raise DomainError(
            code="DOMAIN_ERROR",
            message=f"Cannot use {base} as logarithm base",
            detail="Logarithm base must be positive",
            parameter="base",
        )

    if base == 1.0:
        raise DomainError(
            code="DOMAIN_ERROR",
            message="Cannot use 1 as logarithm base",
            detail="Logarithm base 1 is undefined (division by zero in change-of-base)",
            parameter="base",
        )

    return math.log(value) / math.log(base)
```
